这次死锁的根因，不是两个请求在插入同一条 (post_id, topic_id)，而是“新建草稿时仍然先删关系、再插关系”这条事务路径，在 MySQL 默认 REPEATABLE READ 下把自己绕进了死锁。

从代码链路看，/api/v1/me/drafts 会进入 [ContentController.java](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/)，这里创建草稿时明确传的是 null id，所以一定是“新建帖子”，不是更新旧帖子。随后事务进入 [PostApplicationService.java](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/)，而 save() 在插入 t_post 后，无论新建还是更新，都会调用 [MyBatisPostRepository.java](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 里的 replaceRelations()。这个方法的顺序是固定的：先删图片、先删话题关系，再重新插入；对应到 t_post_topic 就是 [PostTopicMapper.xml](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 的 DELETE FROM t_post_topic WHERE post_id = ?，然后才是 [PostTopicMapper.xml](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 的 INSERT INTO t_post_topic (post_id, topic_id) VALUES (?, ?)。

问题恰恰出在这个“删”。对于新建草稿来说，post_id 是刚生成的自增 id，[V1__init_schema.sql](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 里 t_post.id 是 AUTO_INCREMENT，而 t_post_topic 还不可能有这条草稿的历史关系，所以这次 DELETE ... WHERE post_id = ? 从业务上是空删。但在 InnoDB 默认隔离级别下，空删并不等于“完全不加锁”，它会在 t_post_topic 的 post_id 相关索引范围上拿 gap/next-key lock。接着同一事务又立刻执行 INSERT INTO t_post_topic，而高并发创建时大家的 post_id 都在自增尾部，都会往索引右边界插入，于是很容易形成这条死锁链：

1. 事务 T1 新建帖子，拿到 post_id=1001
2. 事务 T2 新建帖子，拿到 post_id=1002
3. T1 执行 DELETE FROM t_post_topic WHERE post_id=1001，虽然删不到行，但在索引尾部附近留下 gap lock
4. T2 执行 DELETE FROM t_post_topic WHERE post_id=1002，也留下自己的 gap lock
5. T1 开始 INSERT INTO t_post_topic (1001, topic_id)，发现要插入的位置被 T2 的 gap lock 挡住，开始等待
6. T2 开始 INSERT INTO t_post_topic (1002, topic_id)，又被 T1 的 gap lock 挡住
7. 两边互等，MySQL 判定死锁，挑一个受害者，所以异常最终显示在 INSERT 上，而不是前面的 DELETE

这也解释了为什么栈里报错点是 insertTopic，但真正“埋雷”的往往是前一条 deleteByPostId。报错语句是受害者语句，不一定是最早制造锁冲突的语句。

压测场景也正好会把这个问题放大。perf/scenarios/04-content-module.js 的 [04-content-module.js](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 会持续执行“创建草稿 -> 更新草稿 -> 删除草稿”，而混合场景 [07-mixed-dau-15000.js](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 默认还有 35 个 content VU，其中 20% 概率直接创建草稿。也就是说，压测时会有一批并发事务不断重复这条模式：

INSERT t_post -> DELETE t_post_topic WHERE post_id=newId -> INSERT t_post_topic

这条链本身就足以打出死锁，哪怕每个请求只带一个 tag，哪怕不是同一个用户，也哪怕并没有两个请求操作同一个草稿。

一句话总结：这次死锁的本质，是“新建帖子时不必要的关系表清理”在事务里先拿了范围锁，再去插入同一张表的新关系，压测并发下在自增尾部形成了互相等待。

如果你愿意，我下一步可以直接帮你改这块，最有效的修复通常是：

1. 新建帖子时不要调用 deleteByPostId
2. 更新帖子时也尽量不要“全删全插”，改成差量更新
3. 兜底再加一次死锁重试

我是按“从报错点倒推到并发场景”的顺序查的，基本分了四步。

1. 先顺着异常栈还原代码路径。
   我从报错里的 PostTopicMapper.xml 和调用栈往上找，确认请求是从 [ContentController.java](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 的 createDraft() 进来，进入 [PostApplicationService.java](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 的事务方法，再落到 [MyBatisPostRepository.java](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 的 replaceRelations()。这里我先确认了一件事：虽然报错语句是 INSERT INTO t_post_topic，但它前面一定还执行了 DELETE FROM t_post_topic WHERE post_id = ?，所以不能只盯着 insert 看。
2. 再把 SQL 和表结构对起来看。
   我打开了 [PostTopicMapper.xml](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 和 [V3__添加话题相关表.sql](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/)，确认 t_post_topic 有这些关键信息：
   INSERT INTO t_post_topic (post_id, topic_id) VALUES (?, ?)
   DELETE FROM t_post_topic WHERE post_id = ?
   还有唯一索引 (post_id, topic_id)、普通索引 (post_id)、联合索引 (topic_id, post_id)。
   看到这里我就知道，锁竞争大概率发生在 post_id 相关索引范围上，而不是业务层面“同一篇帖子重复加同一话题”。
3. 然后去核对压测请求到底在怎么打。
   我读了压测脚本 [04-content-module.js](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 和 [07-mixed-dau-15000.js](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/)。这里的关键发现是：
   draftLifecycle 会持续做“创建草稿 -> 更新草稿 -> 删除草稿”。
   混合场景里还有不少并发用户会随机创建草稿。
   也就是说，压测不是偶发地改某一条草稿，而是在持续并发跑“新建帖子并写 t_post_topic”这条路径。这个发现很重要，因为它让我把怀疑点从“同一 post_id 冲突”转向了“新建时索引尾部竞争”。
4. 最后确认是不是 InnoDB 的典型锁模式。
   我又查了 [V1__init_schema.sql](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/) 和 [Post.java](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/)，确认 t_post.id 是 AUTO_INCREMENT。再看配置 [application.yml](https://file+.vscode-resource.vscode-cdn.net/c%3A/Users/hanjie/.vscode/extensions/openai.chatgpt-26.5406.40811/webview/)，项目里没有显式改事务隔离级别，所以我按 MySQL 默认 REPEATABLE READ 来分析。
   这样链路就闭环了：新建草稿时拿到递增的 post_id，但代码还是先执行一次“删旧关系”。这条 DELETE 对一个新 post_id 虽然删不到数据，却可能在索引范围上加 gap/next-key lock；后面再 INSERT 新关系时，多个事务就会在索引右边界互相等待，最后由 MySQL 在某条 INSERT 上报死锁。

我实际用的办法很朴素，就是先用 rg 全局搜调用链和 mapper，再逐个读关键文件，把“异常栈、事务边界、SQL 顺序、索引结构、压测模型”这五件事拼在一起。死锁排查里，单看报错 SQL 往往会误判，必须把它放回事务上下文里看。
