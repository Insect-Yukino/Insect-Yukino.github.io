为什么 `Integer[]` 性能显著低于 `int[]`？

`Integer[]` 因为**对象间接访问 + 内存离散分布**，破坏了 CPU 缓存局部性，导致 cache miss 增多，性能远低于 `int[]`。

`Integer[]` 中存的是 `Integer` 对象的引用。这使得对象分配十分的零散。反观 `int[]`，他在内存中就是一块连续的内存数组。为什么连续空间的 `int[]` 性能优于 `Integer[]`。

这是由于 CPU 的设计。

CPU 不直接从内存读数据，它通过

```text
内存 → L3 → L2 → L1 → 寄存器
```

而且他不是一次读取四个字节，而是读取一整块 Cache Line。这叫空间局部性。

因此 `int[]` 被分配在一整块地址，所有 `int` 连续存储，每个 4 字节，一条 Cache Line 可以装 16 个 `int`。遍历时，CPU 预取机制生效，缓存全部命中。

反观 `Integer[]` 的内存布局，`Integer` 的引用被分配在了一个连续的内存空间中，真正的数据却在堆里，于是访问一个元素变成了

1. 读引用
2. 根据引用跳转
3. 再读对象

`Integer[]` 中 `Integer` 对象不保证连续，可以出现在堆的任意位置，CPU 预读失败导致效率低下。

同时产生的问题还有 GC 和内存开销变大。

我们只要随便 `new` 一个对象

```java
Object obj = new Object();
```

就会产生 12~16 字节头对象，而 `int` 只有 4 个字节。
