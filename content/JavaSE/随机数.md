今天在力扣中刷到了通过 `Rand7()` 实现 `Rand10()` 的题，了解一下随机数。

随机数在计算机里大致分两类：**伪随机** 和 **真随机**。

### 1) 计算机里的随机数通常怎么实现？

本质：**一个确定性的算法 + 一个种子 seed**，生成一串看起来随机的数。

只要 **seed 一样**，输出序列就 **完全一样**（可复现），生成随机数时，速度快、好用、方便测试。但它不是真正不可预测的随机，因为算法是确定的。

### 2) 什么是伪随机数？

伪随机指的是：

**统计性质上像随机**（均匀、分布看起来对），但 **本质上可由算法预测/重现**（知道 seed 或内部状态就能推算后续）。

所以伪随机满足够用的随机，但不一定满足安全的随机。

### 3) 真随机数

真随机数不是算法算出来的，而是从 **物理噪声** 里采样，例如：

- 热噪声、电子噪声
- 晶振抖动（jitter）
- 放射性衰变
- 光子计数等

特点：

- **不可预测**（即使知道采样方式也难预测下一位）
- 但通常 **慢**、**需要硬件**、**还要做去偏差/熵提取**（因为物理源可能有偏）

很多 CPU/系统也会提供硬件随机源或熵源：

- Linux 的 `/dev/random`、`/dev/urandom`（来自系统熵池）
- 一些 CPU 指令（不同厂商不同实现）

### 4) 加密安全伪随机

加密安全伪随机：仍然是算法，但设计目标是：

- 即使攻击者看到很多输出，也很难反推出内部状态或预测未来输出

实现通常来自：

- 操作系统熵池 + 加密算法（如 AES-CTR / ChaCha20）
- 例如 Java 的 `SecureRandom`

### `java.util.Random`：经典伪随机

`Random` 是最传统的实现，内部是 **48 位种子 + 线性同余算法（LCG）**，每次根据公式更新 seed，然后从 seed 里取高位输出。Oracle/OpenJDK 的文档明确说了它用 48-bit seed，并用线性同余公式更新。 

特点：

- **快、可复现**（同样 seed 会得到同样序列）
- **不适合安全场景**（可预测）
- 在并发下共享一个 `Random` 会有争用/性能问题（通常建议用 ThreadLocalRandom）

### `ThreadLocalRandom`：并发友好的伪随机

它本质还是伪随机数，但设计成 **每个线程维护自己的状态**，避免多线程抢同一个 `Random` 的锁/原子更新导致性能掉。

适用：高并发下做抽样、打散、随机延迟等非安全用途。

### `SplittableRandom`：面向并行/流式计算的伪随机

`SplittableRandom` 主要为并行计算设计：支持 `split()` 生成互不共享状态的新生成器，适合 ForkJoin/并行流这种场景。它是伪随机，并强调可拆分、低相关性与性能。 

适用：并行任务里的 Monte Carlo、采样、洗牌等（仍然**不是安全随机**）。

### Java 17+：`RandomGenerator` 与更多算法

从 Java 17 开始，JDK 增强了随机数 API（JEP 356）：提供新的接口与多种实现，让我们可以按需求选择不同为随机算法，并且 **保留 `Random` 的旧行为不变**。 

适用：想明确选择算法、追求更好的统计性质/并行特性时。

### `java.security.SecureRandom`：加密安全伪随机

`SecureRandom` 仍然是伪随机的范畴（算法生成），但它的目标是：**输出不可预测**，满足安全用途。

它怎么做到更像真随机？

- `SecureRandom` 会从 **操作系统的熵源** 获取不可预测的随机性（鼠标键盘中断、硬件噪声、系统熵池等）来 **播种/补充熵**，然后用加密强度的算法扩展为大量随机输出。 
- 这样即使你生成很多随机数，也很难反推出内部状态或预测未来（前提是使用得当）。

适用：密钥、nonce、token、验证码、抽奖防预测、任何安全相关。
