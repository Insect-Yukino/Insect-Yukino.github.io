文件拷贝操作，应该就是我们俗称的 IO 操作，典型的流程是：

```text
磁盘 -> 内核缓冲区 -> 用户缓冲区 -> 内核缓冲区 -> 目标磁盘
```

这样会发生多次数据拷贝和用户态和内核太的切换。

现代操作系统会通过零拷贝优化，流程变成 
```text
磁盘 -> 内核缓冲区 -> 目标磁盘
```

这样，IO 操作会更快，更少占用 CPU 资源，更少内存拷贝。

我们使用 Java 进行文件拷贝操作时，调用的是操作系统层的能力。

传统的方式是：

```java
InputStream in = new FileInputStream(src);
OutputStream out = new FileOutputStream(dest);

byte[] buffer = new byte[8192];
int len;
while ((len = in.read(buffer)) != -1) {
    out.write(buffer, 0, len);
}
```

他会经过：

```text
磁盘 → 内核 → JVM 用户内存 → 内核 → 磁盘
```

涉及到多次内存拷贝，性能一般。

而 NIO FileChannel 使用操作系统底层的零拷贝技术

```java 
FileChannel source = new FileInputStream(src).getChannel();
FileChannel target = new FileOutputStream(dest).getChannel();

source.transferTo(0, source.size(), target);
```

`transferTo()` 在底层会调用：

- Linux → `sendfile`
- Windows → `TransmitFile`

涉及到的路径就变成了

```text
磁盘 → 内核 → 目标磁盘
```

大大减少了状态切换

更加推荐的是使用 `Files#copy`

```java
Files.copy(srcPath, destPath);
```

他的底层可能使用流，或者使用 channel 或者调用系统优化方法。JDK 会根据平台自动优化。

IO 操作是系统占用资源最多的操作之一，为什么呢？

我的理解是，操作系统会为每个进程分配 CPU 时间片，如果当前线程分配了时间片，但是却不使用 CPU，IO 操作造成了进程阻塞，这就白白浪费了 CPU 资源。

操作系统对 IO 操作相关方面做了一系列优化。

### 页缓存（Page Cache）

拷贝文件经常不需要真正读磁盘→写磁盘每次都碰盘：

- 读取的数据会进 page cache
- 写入也先写到 page cache（脏页），后台再落盘

所以看起来复制很快，并且顺序 I/O 吞吐很高。

### 零拷贝

尤其是文件→网络的典型链路对比：

**传统方式**

内核读文件 → 拷贝到用户态 buffer → 再写回内核 socket buffer → 网卡

**`sendfile`**

内核直接把文件页缓存的数据送到 socket → 网卡。用户态几乎不搬运数据，CPU 占用低，吞吐更稳定。

### Copy-on-write / Reflink

在支持的文件系统上（如 Linux 的 btrfs/xfs reflink、macOS APFS clone、Windows ReFS 某些场景），**文件复制可以变成元数据操作**：先共享同一份数据块，后续改动才复制（COW）。

### 内核里的大块拷贝 + 向量化

就算不是零拷贝，内核也会用更大的块、更少的 syscall、更好的预读/写回策略，整体比在 Java 里 while 循环 8KB/16KB 复制更容易跑满吞吐。

什么时候自己循环拷贝反而差

- buffer 太小（8KB/16KB）→ syscall 次数多
- 频繁 flush/fsync → 把顺序写打碎
- 同时做 CPU 重活（压缩/加密/生成内容）→ 让 socket 写不连续，易触发超时
- 从网络流复制到文件时用同步阻塞 + 小块，吞吐波动更大
