在实际项目里，通常把异常分成**系统异常**和**业务异常**。它们的本质区别是：

- **业务异常**：业务规则不满足，但系统本身是健康的、可预测的。
- **系统异常**：通常是系统/依赖/代码出问题了，是非预期的。

异常体系分层

```text
Controller / Service
   │
   ├── 主动抛出 → 业务异常（BizException）
   │
   └── 被动抛出 → 系统异常（RuntimeException / Exception）
                     ↓
            统一异常处理器（收敛 & 翻译）
```

> 核心原则：所有异常都抛，但只有业务异常是设计出来的，系统异常是被动发生、由框架兜底收敛的结果。

## 业务异常

业务异常代表用户/请求触发了业务规则的拒绝，系统通过校验后主动抛出，属于预期内。常见的场景有：库存不足、余额不足等。

当系统抛出业务异常时因遵循以下处理原则：

- 不要当成 bug，不打印 ERROR 堆栈（最多 warn/info，带业务码）
- 给用户明确可理解的提示
- 统一错误码
- 不触发告警
- 最好可追踪

BizException 示例

```java
public class BizException extends RuntimeException {

    private final ErrorCode errorCode;

    public BizException(ErrorCode errorCode) {
        super(errorCode.getMessage());
        this.errorCode = errorCode;
    }

    public BizException(ErrorCode errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    public ErrorCode getErrorCode() {
        return errorCode;
    }
}
```

业务码定义

```java
public enum ErrorCode {

    SUCCESS("SUCCESS", HttpStatus.OK, "成功"),

    // 参数
    PARAM_INVALID("PARAM_INVALID", HttpStatus.BAD_REQUEST, "参数不合法"),

    // 认证 & 授权
    USER_NOT_LOGIN("USER_NOT_LOGIN", HttpStatus.UNAUTHORIZED, "未登录"),
    PERMISSION_DENIED("PERMISSION_DENIED", HttpStatus.FORBIDDEN, "无权限"),

    // 订单
    ORDER_NOT_FOUND("ORDER_NOT_FOUND", HttpStatus.NOT_FOUND, "订单不存在"),
    ORDER_STOCK_NOT_ENOUGH("ORDER_STOCK_NOT_ENOUGH", HttpStatus.CONFLICT, "库存不足"),

    // 系统
    SYSTEM_ERROR("SYSTEM_ERROR", HttpStatus.INTERNAL_SERVER_ERROR, "系统繁忙，请稍后再试");

    private final String code;
    private final HttpStatus httpStatus;
    private final String message;

    // constructor + getter
}
```

> 业务异常是否应该纠结 HTTP 状态码？
>
> 业务异常的 HTTP 状态码选择，本质是**接口风格问题，而不是异常类型问题**：
>
> - 如果系统偏 RPC / 国内 BFF 风格：常用 **HTTP 200 + 业务码**
> - 如果系统偏 REST：更推荐 **4xx + 业务码**
>
> 关键点不在于状态码，而在于：**业务异常必须返回稳定、可判断的业务码（code）**

## 系统异常

系统异常代表系统无法按设计运行，属于非预期故障，需要排查修复或降级。

常见场景有：

- NPE、数组越界、类型转换异常（代码 bug）
- 数据库连接超时、死锁、慢查询导致超时
- Redis/MQ/第三方接口不可用或超时
- 线程池打满、内存溢出、磁盘满
- 配置错误、序列化失败

系统异常的特点是：**不可预期、需要日志堆栈、需要告警、需要修复或降级**。

系统异常的处理原则是：

- **统一兜底**：任何未捕获异常最终都要被全局异常处理器兜住

- **记录完整堆栈**（error 级别）+ 关键上下文（traceId、参数摘要、用户信息）

- **对外返回通用错误**：不要把堆栈/SQL/内部地址暴露给前端

- **触发告警**：尤其是依赖不可用、错误率上升、超时

- **降级/重试/熔断**

- **幂等保证**：系统异常发生在关键写操作时要能回滚或保证不重复写

## 业务码设计

业务码决定了系统长期的可维护性问题。一个好的业务码需要满足：

- **稳定不变**（一旦发布，永不修改）
- **机器可读**（前端 / 网关 / 调用方可判断）
- **人可理解**（看到码就知道大概是哪类问题）
- **可扩展**（不会 1 年后用完）
- **能定位责任域**（是订单问题？库存？权限？）

> 要明确业务码、message 和 HTTP 的状态码的作用：
>
> - 业务码用来给程序做判断用 `if/switch`
> - message 是给人看的提示文案
> - HTTP 状态码是给协议/网关/中间件看的

### 业务码的分类设计

#### 第一层错误类型

```text
SUCCESS
PARAM_ERROR
BIZ_ERROR
SYS_ERROR
```

#### 模块维度

```text
USER_
ORDER_
CART_
STOCK_
PAY_
COUPON_
COMMENT_
```

### 具体业务含义

```text
ORDER_NOT_FOUND
ORDER_ALREADY_PAID
ORDER_STOCK_NOT_ENOUGH
ORDER_STATUS_INVALID
```

最终前端拿到业务码，做逻辑处理

```js
if (res.code === 'USER_NOT_LOGIN') {
  redirectToLogin();
}

if (res.code === 'ORDER_STOCK_NOT_ENOUGH') {
  showToast(res.message);
}
```

强烈推荐使用字符串作为业务码，如：

```text
ORDER_STOCK_NOT_ENOUGH
ORDER_STATUS_NOT_ALLOW_CANCEL
PAY_AMOUNT_MISMATCH
USER_NOT_LOGIN
PERMISSION_DENIED
```

这样的优点是：

- 可读性极强

- 不需要查表

- 不怕位数不够

- 不怕跨系统冲突

- 日志/告警/排查友好

使用数字业务码的维护成本相对较高

```text
1001 01 01 001
```

- 1001：系统（电商）

- 01：订单模块

- 01：业务校验失败

- 001：库存不足

## 在 Spring 中如何统一处理异常

一般来说，在业务层，我们只需负责判断 + 抛异常的工作，抛出后的异常最终由统一异常处理器处理。

统一异常处理的必要性体现在保持代码的一致性、提供更清晰的错误信息、以及更容易排查问题。通过定义统一的异常处理方式，确保在整个应用中对异常的处理保持一致，减少了重复编写相似异常处理逻辑的工作，同时提供友好的错误信息帮助开发者和维护人员更快地定位和解决问题，最终提高了应用的可维护性和可读性。

### Spring MVC 如何进行异常处理

我们一般定义一个全局异常处理类处理业务异常和系统异常

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    // 业务异常
    @ExceptionHandler(BizException.class)
    public ResponseEntity<ApiResponse<Void>> handleBizException(BizException e) {

        ErrorCode code = e.getErrorCode();

        log.warn("BizException: code={}, msg={}", code.getCode(), e.getMessage());

        return ResponseEntity
                .status(code.getHttpStatus())
                .body(ApiResponse.fail(code.getCode(), e.getMessage()));
    }

    // 参数校验异常
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiResponse<Void>> handleValidException(MethodArgumentNotValidException e) {

        String msg = e.getBindingResult()
                      .getFieldErrors()
                      .stream()
                      .map(FieldError::getDefaultMessage)
                      .findFirst()
                      .orElse("参数不合法");

        return ResponseEntity
                .status(HttpStatus.BAD_REQUEST)
                .body(ApiResponse.fail("PARAM_INVALID", msg));
    }

    // 系统异常兜底
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse<Void>> handleSystemException(Exception e) {

        log.error("SystemException", e);

        return ResponseEntity
                .status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(ApiResponse.fail(
                        ErrorCode.SYSTEM_ERROR.getCode(),
                        ErrorCode.SYSTEM_ERROR.getMessage()
                ));
    }
}

```

ApiResponse 统一返回体

```java
@Data
@AllArgsConstructor
@NoArgsConstructor
public class ApiResponse<T> {

    private String code;
    private String message;
    private T data;

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>("SUCCESS", "成功", data);
    }

    public static <T> ApiResponse<T> fail(String code, String message) {
        return new ApiResponse<>(code, message, null);
    }
}
```

`@RestControllerAdvice` 用于定义全局异常处理类。`@ExceptionHandler`注解用于在这个类中定义异常对应的处理逻辑。`@ExceptionHandler` 也可以在 Controller 控制器类中使用，定义的方式和全局异常处理器中的方式相同，但是作用域会被限制在该控制器中。

```java
@RestController
public class TestController {
    @RequestMapping("/test")
    public ResultResponse test() {
        throw new ServiceException();
    }
    
    @ExceptionHandler(ServiceException.class)
    public ResultResponse handleServiceException(ServiceException e) {
        
    }
}
```

#### 异常是从哪里开始被 Spring MVC 接管的？

> 是否好奇，Controller 控制器中抛出的异常是如何被 Spring 框架接管的。

当请求进入 Spring MVC 后，核心入口是 `DispatcherServlet#doDispatch(...)`。
 然后，该请求会：

1. 找到 Handler（Controller 方法）
2. 调用 Handler（通过 HandlerAdapter）
3. 渲染返回值（视图 or JSON）

**异常的拦截点在第 2 步：执行 Handler 的过程中**。

伪流程：

```
DispatcherServlet.doDispatch
  ├─ handler = getHandler(request)
  ├─ try
  │    ├─ mv = handlerAdapter.handle(request, response, handler)  // 调用 Controller
  │    └─ processDispatchResult(mv)                               // 正常渲染
  └─ catch (Exception ex)	// 如果捕获了异常
       └─ processHandlerException(ex)  // 进入异常处理链
```

**一旦 Controller/参数绑定/拦截器等抛出异常，就会进入 `processHandlerException`。**

#### `HandlerExceptionResolver` 是什么？

上文说到的异常链就是由 `HandlerExceptionResolver` 组成的。

`HandlerExceptionResolver` 可以理解为：

> **把异常翻译为 HTTP 响应（或 ModelAndView）的组件。**

Spring MVC 会维护一个 **resolver 列表**，抛出的异常会按顺序尝试：

- 哪个 resolver 能处理异常，就用它生成响应
- 一旦处理成功，就停止后续 resolver

#### 默认的三个 Resolver 分别负责什么？

在 Spring Boot 默认配置里，典型顺序就是：

1. `ExceptionHandlerExceptionResolver`
2. `ResponseStatusExceptionResolver`
3. `DefaultHandlerExceptionResolver`

下面逐个讲清楚它们的**匹配规则、能处理什么、处理结果是什么**。

##### `ExceptionHandlerExceptionResolver`

它解决的问题是，让异常处理器中的 `@ExceptionHandler` 生效（全局或局部）。

它会去找两类地方：

1. **当前 Controller 内**有没有 `@ExceptionHandler` 方法
2. **全局 `@ControllerAdvice / @RestControllerAdvice`** 里有没有能处理的 `@ExceptionHandler`

它如何判断能不能处理？

- `@ExceptionHandler` 注解声明的异常类型是否匹配当前异常
- 允许父子类匹配：比如 `@ExceptionHandler(RuntimeException.class)` 可以兜住 `NullPointerException`
- 匹配越精确的方法优先（子类优先于父类）

处理成功后

- 对 REST：返回构造的 `ResponseEntity` / JSON（通常 `@RestControllerAdvice`）
- 对 MVC：返回 `ModelAndView`

> **工程上写的统一异常处理器，就是依赖它来生效的。**

##### `ResponseStatusExceptionResolver`（声明式状态码）

它让 `@ResponseStatus` 或 `ResponseStatusException` 这种声明式异常生效。

它主要识别两种：

A. 异常类上标注了 `@ResponseStatus`

```java
@ResponseStatus(HttpStatus.NOT_FOUND)
class OrderNotFoundException extends RuntimeException {}
```

B. 直接抛 `ResponseStatusException`

```java
throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "参数错误");
```

处理成功后，它会设置：

- HTTP 状态码（例如 404/400）
- 可选 reason（但一般 REST 项目更推荐用统一 JSON）

**注意**：如果项目中已经用 `@RestControllerAdvice` 统一返回 JSON，通常不建议大量用 `@ResponseStatus`，避免风格混杂。

##### `DefaultHandlerExceptionResolver`

它解决的问题没有处理到的 Spring MVC 内部的一些常见异常。

它处理的多是框架层异常，比如：

- `HttpRequestMethodNotSupportedException` → 405（方法不支持）
- `HttpMediaTypeNotSupportedException` → 415（Content-Type 不支持）
- `MissingServletRequestParameterException` → 400（缺少参数）
- `TypeMismatchException` → 400（参数类型转换失败）

这些很多发生在：

- 参数绑定/校验阶段（Controller 方法还没真正执行）
- 请求方法/媒体类型不匹配阶段

> **它的价值是：什么都不写，也不会所有错误都变 500。**

#### 都没处理会怎样？为什么会走 `/error`？

当这串 resolver 都处理不了异常时，Spring MVC 就无法把异常转换成响应。

此时，尤其在 Spring Boot，会走到默认错误处理机制：

- 请求会被导向 `/error`
- 由 `BasicErrorController` 输出结果（JSON 或 HTML）

可以把 `/error` 理解为：

> **最后的兜底：程序中没接住的异常，交给它给一个默认错误响应。**

`BasicErrorController` 读的是 `ErrorAttributes`，默认 JSON 常见字段：

```json
{
  "timestamp": "...",
  "status": 500,
  "error": "Internal Server Error",
  "path": "/xxx"
}
```

如果在全局异常处理器里写了：

```java
@ExceptionHandler(Exception.class)
public ResponseEntity<?> handle(Exception e) { ... }
```

那么：

- 绝大多数异常都会被**第一层 `ExceptionHandlerExceptionResolver`** 捕获并处理
- 后面的 `ResponseStatusExceptionResolver` / `DefaultHandlerExceptionResolver` 很多情况下就不会再发挥作用
- `/error` 也基本不会走到

**这通常是我们想要的效果**（统一返回体），但前提是：

- 要区分业务异常 vs 系统异常
- 系统异常要打完整堆栈 + 告警
- 业务异常不要 error 堆栈

> 读完上面的内容，我们清楚了。Spring MVC 框架的异常处理体系。我们最长使用的就是 `RestControllerAdvice`。

## Spring Security 的异常体系

如果在项目中集成了 Spring Security 的框架，在 Spring Security 抛出的异常是不会被 Spring MVC 处理的，因为这些异常通常是 Web Filter 级别的，发生在 Spring MVC 之前。

在 Spring Security 中，异常不是随便抛的，它们高度结构化，主要集中在两大类：

### 认证相关异常

这些异常共同的父类是：

```java
AuthenticationException
```

最常见的有：

| 异常类                                       | 含义                | 常见场景                     |
| -------------------------------------------- | ------------------- | ---------------------------- |
| `AuthenticationCredentialsNotFoundException` | 没有 Authentication | 未登录访问受保护资源         |
| `BadCredentialsException`                    | 凭证错误            | 用户名/密码错误              |
| `InsufficientAuthenticationException`        | 认证信息不足        | token 失效、remember-me 失效 |
| `AccountExpiredException`                    | 账号过期            | 账号到期                     |
| `LockedException`                            | 账号被锁            | 风控/后台禁用                |
| `DisabledException`                          | 账号被禁用          | 用户被封号                   |
| `CredentialsExpiredException`                | 凭证过期            | 密码过期                     |
| `JwtException`（或其子类）                   | JWT 非法            | token 签名错误、过期         |

只要是 AuthenticationException，最终一定走 401

### 授权相关异常

核心类

```java
AccessDeniedException
```

典型场景是：已登录，token 有效，前端访问了 `@PreAuthorize("hasRole('ADMIN')")` 的接口。但是当前用户没有 ADMIN 权限。

401 / 403 在 Spring Security 中的核心判断规则

```text
是否存在有效 Authentication？
    ├── 否 → 401（AuthenticationEntryPoint）
    └── 是
         ├── 权限不够 → 403（AccessDeniedHandler）
         └── 允许访问 → 放行
```

在代码中，由 `ExceptionTranslationFilter` 来实现。它做的事可以概括为，将异常翻译成 HTTP 语义。简化后的代码逻辑：

```java
try {
    chain.doFilter(request, response);
} catch (AuthenticationException ae) {
    // 未认证
    SecurityContextHolder.clearContext();
    authenticationEntryPoint.commence(request, response, ae);
} catch (AccessDeniedException ade) {
    if (isAnonymousOrRememberMe()) {
        // 看起来像未登录
        authenticationEntryPoint.commence(
            request, response,
            new InsufficientAuthenticationException("Full authentication is required", ade)
        );
    } else {
        // 已登录但没权限
        accessDeniedHandler.handle(request, response, ade);
    }
}
```

Spring Security 处理以上异常的关键配置：

```java
http
  .csrf(csrf -> csrf.disable())
  .sessionManagement(sm -> sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
  .authorizeHttpRequests(auth -> auth
      .requestMatchers("/auth/**", "/public/**").permitAll()
      .anyRequest().authenticated()
  )
  .exceptionHandling(ex -> ex
      .authenticationEntryPoint(authenticationEntryPoint())
      .accessDeniedHandler(accessDeniedHandler())
  );
```

异常处理类通过 Bean 注入 `AuthenticationEntryPoint`，`AccessDeniedHandler` 类

```java
@Bean
AuthenticationEntryPoint authenticationEntryPoint() {
    return (req, res, ex) -> {
        res.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
        res.setContentType("application/json;charset=UTF-8");
        res.getWriter().write("""
            {"code":"USER_NOT_LOGIN","message":"未登录或令牌无效"}
        """);
    };
}

@Bean
AccessDeniedHandler accessDeniedHandler() {
    return (req, res, ex) -> {
        res.setStatus(HttpServletResponse.SC_FORBIDDEN);
        res.setContentType("application/json;charset=UTF-8");
        res.getWriter().write("""
            {"code":"PERMISSION_DENIED","message":"没有访问权限"}
        """);
    };
}
```

自定义的 JWT Filter 中可能抛出的异常应该也纳入 Spring Security 异常体系

```java
try {
    // 解析 JWT
} catch (JwtException e) {
    // 抛 AuthenticationException
    throw new InsufficientAuthenticationException("Invalid token", e);
}
```
