进程切换有什么开销？

进程切换的开销主要来自 **把一个进程的运行现场保存起来，再把另一个进程的现场恢复起来，以及随之而来的缓存/内存/调度副作用**。常见开销点如下：

#### 1）CPU 上下文保存/恢复

- 保存/恢复寄存器（通用寄存器、程序计数器 PC、栈指针 SP、标志寄存器等）
- 可能还包括 SIMD/FPU 寄存器（取决于系统是否延迟保存）

这部分是必付的固定成本

#### 2）内核态切换与调度成本

进程切换一定会进入内核：

- 触发调度器：选择下一个运行的进程/线程
- 更新运行队列、优先级、时间片等结构
- 加锁/解锁（多核下尤其明显）

#### 3）TLB 失效与页表相关开销（进程切换比线程切换更重）

进程通常有不同的地址空间：

- 切换会导致 **页表基址（如 CR3）变化**
- 结果是 **TLB（地址翻译缓存）命中率下降**，甚至需要刷新/部分失效
- 后续内存访问会频繁走页表遍历，延迟显著上升

> 线程切换通常共享同一地址空间，因此一般不会触发这种程度的 TLB 代价（除非跨进程线程或某些隔离机制）。

#### 4）CPU Cache 污染（L1/L2/L3 命中率下降）

切换到另一个进程后：

- 之前进程的热数据/指令不再被使用，缓存被新进程逐步覆盖
- 造成 **指令缓存 I-Cache**、**数据缓存 D-Cache**、甚至 **LLC** 命中率下降
- 这往往是最隐形但最贵的部分：不是切换瞬间贵，而是切换后的一段时间都变慢

#### 5）分支预测/流水线扰动

新进程的代码路径不同：

- 分支预测历史失效，预测命中率下降
- CPU 流水线更频繁 flush / stall
- 典型表现：短时间内 IPC 下降

#### 6）NUMA / CPU 亲和性导致的远端内存访问

如果进程被调度到不同 CPU 核甚至不同 NUMA 节点：

- 内存可能变成远端访问，延迟更高
- 缓存局部性进一步变差
- 所以生产上会强调 CPU affinity、NUMA awareness

#### 7）如果涉及阻塞/唤醒：还会有额外成本

很多切换发生在：

- 系统调用阻塞（IO、锁竞争）
- 中断唤醒
- futex / mutex 等同步原语

此时还会叠加：

- 线程/进程睡眠队列管理
- 竞争导致的抖动（thundering herd）

**进程切换通常比线程切换更贵**，核心差异在 **地址空间切换 → TLB/页表代价**，以及更强的 cache 冲击。真正让系统慢的，往往不是那几百纳秒/几微秒的保存恢复，而是 **TLB + Cache 命中率下降带来的后续访问变慢**。
