单核 CPU 的瓶颈本质就是：**同一时刻只能执行一条指令流**。当程序需要做的每个请求要消耗的 CPU 指令数 × 请求数超过这一核在单位时间内能执行的指令数，就会顶满，表现为 **CPU 100%**、延迟飙升、吞吐不再增长。

#### 1) 单核 CPU 的瓶颈通常卡在哪些地方？

从工程角度看，单核的上限主要由这几类决定：

1. **纯计算量**
    每次请求要做解析、分支判断、哈希、拷贝、编码/解码、压缩、加密等。请求越多越容易满。
2. **内存访问 / 内存带宽**
    很多服务不是算不过来，而是等内存。比如大量随机访问哈希表、跳表、对象分配和拷贝，CPU 在等数据从缓存/内存回来。
3. **系统调用与网络 I/O 的开销**
    `read/write/epoll`、包拷贝、协议栈处理、上下文切换等也会吃 CPU（尤其高并发小包）。
4. **单线程的排队效应**
    只要有一个请求执行得慢（比如 O(N) 大操作），后面所有请求都在它后面排队，**尾延迟** 会很明显。

> 所以单核瓶颈不等于算力不够，也可能是内存/网络/系统开销先把 CPU 用完了。

#### 2) Redis 单线程 = 只能用一个核吗？

要分清两件事：

- Redis 的核心命令执行模型：基本是单线程

- Redis 采用（基本）单线程执行命令的设计，优势是：
  - **避免锁竞争**（数据结构操作不需要加锁）
  - **上下文切换少**
  - 延迟更稳定，代码复杂度低

但 Redis 不是全程只有一个线程，现代 Redis 会有一些后台线程/子进程做：

- 持久化（RDB/AOF rewrite 用 fork 子进程）
- 异步释放内存、后台清理等
- Redis 6+ 支持 **I/O threading**
   但注意：**命令执行（对数据结构的实际操作）仍然主要在主线程**。

所以严格说：**一个 Redis 实例的数据操作吞吐主要受单核限制**，但不代表 Redis 完全只用一个核做所有事。

#### 3) 那 Redis 不会遇到 CPU 瓶颈吗？

会的。只是很多业务场景下它 **很难先被命令执行 CPU 卡死**，原因是：

1. **Redis 常见命令很“轻”**
    GET/SET/HGET/HSET/INCR 这种操作通常是 O(1) 且路径短，单核能扛很高的 QPS（具体多高取决于 CPU、网络、数据大小、客户端模式等）。
2. **很多时候先卡在网络/协议/拷贝**
    小 value、超高 QPS 时，Redis 可能更接近网络 I/O + 协议解析瓶颈，而不是复杂计算。
3. **单线程避免锁，效率反而更高**
    多线程共享数据结构往往要加锁，锁竞争可能吃掉大量 CPU，使吞吐并不线性增长。Redis 用单线程绕开了这类损耗。

但以下情况，Redis **非常容易 CPU 100%** 或延迟暴涨：

- **大 key / 大 value**：一次 GET/SET 就要拷贝很多字节、编码解码、网络发送都重
- **O(N) 或阻塞型命令**：如 `KEYS`、大范围 `LRANGE`、大 `SMEMBERS`、大 `SORT`、`SUNION` 等（取决于规模）
- **Lua 脚本/模块做了重计算**：脚本在主线程跑，跑久了其它请求都得排队
- **高频过期淘汰 + 大量对象创建释放**：内存管理开销上来
- **慢查询导致排队效应**：即使 QPS 不高，尾延迟也会很难看

#### 4) Redis 只能用单核怎么扩展到多核？

Redis 的常见扩展方式是 **横向扩展**，而不是把一个实例做成多核并行执行命令：

1. **多实例**（同一台机器起多个 Redis 进程/实例，绑不同端口）
    让每个实例吃一个核，简单直接。
2. **Redis Cluster / 分片**
    把 key 空间分到多个节点/实例，自然用上多核、多机。
3. **读写分离**（主从复制，读走 replica）
    适合读多写少，但要注意一致性需求。
4. **I/O threading + pipeline**（降低网络/系统调用开销）
    pipeline 能显著减少 RTT/系统调用次数，让单核更“值钱”。
