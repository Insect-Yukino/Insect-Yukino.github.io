这是一个**操作系统里非常核心、也非常容易被误解的概念**。
 我会按 **定义 → 创建方式 → 数据关系 → 真实共享手段 → 常见误区** 的顺序，把**父子进程**一次讲透。

------

## 一、什么是父进程 / 子进程？

### 1️⃣ 定义（操作系统视角）

- **父进程（Parent Process）**：
   👉 创建其他进程的进程
- **子进程（Child Process）**：
   👉 被父进程创建出来的新进程

在类 Unix 系统（Linux / macOS）中：

- 几乎所有进程（除 `init/systemd`）都有父进程
- 进程之间形成一棵 **进程树**

------

### 2️⃣ 父子进程是如何产生的？

在 Linux 中，最经典的方式是：

```
pid_t pid = fork();
```

- `fork()` 调用一次
- 返回两次：
  - 在父进程中返回 **子进程 PID**
  - 在子进程中返回 **0**

```
父进程
  |
  +-- fork()
       |
       +-- 子进程
```

------

## 二、父子进程最重要的结论（先给结论）

> ❗ **父子进程在逻辑上“有关联”，
>  但在内存上“相互隔离”**

这是理解一切问题的核心。

------

## 三、父子进程会共享数据吗？（重点）

### ✅ 结论一句话

> **父子进程默认不共享内存数据**

------

### 1️⃣ fork 之后发生了什么？

当父进程 `fork()` 出子进程时：

- 子进程会得到：
  - 父进程 **虚拟地址空间的拷贝**
  - 父进程的代码段、数据段、堆、栈的“映射”
- 但这是：
  - **虚拟内存层面的拷贝**
  - **不是同一块物理内存**

------

### 2️⃣ Copy-On-Write（写时复制）机制（非常关键）

操作系统为了效率，采用 **COW（写时复制）**：

- fork 后：
  - 父子进程最开始 **共享同一份物理内存页（只读）**
- 当 **任意一方尝试写内存**：
  - OS 才真正复制一份内存页
  - 两边各自修改，互不影响

👉 所以你会看到：

- **读是“看起来共享”**
- **写立刻隔离**

------

### 3️⃣ 举个直观例子

```
int x = 10;
pid_t pid = fork();

if (pid == 0) {
    x = 20;
    printf("child: %d\n", x);
} else {
    sleep(1);
    printf("parent: %d\n", x);
}
```

输出一定是：

```
child: 20
parent: 10
```

👉 因为写操作触发了写时复制。

------

## 四、那父子进程到底“共享”什么？

虽然**不共享普通内存变量**，但会**继承一些内核资源的引用**。

### ✅ 会“继承 / 共享”的东西

| 资源         | 是否共享 | 说明               |
| ------------ | -------- | ------------------ |
| 代码段       | 逻辑相同 | 程序代码一致       |
| 虚拟地址布局 | 相同     | 地址值看起来一样   |
| 文件描述符   | ✅ 共享   | 指向同一个打开文件 |
| 当前工作目录 | ✅ 继承   | fork 时复制        |
| 环境变量     | 复制     | 之后互不影响       |
| 信号处理方式 | 继承     | 可独立修改         |

------

### 🔥 文件描述符共享（重点）

```
int fd = open("a.txt", O_WRONLY);
fork();
```

- 父子进程：
  - 都持有 `fd`
  - 指向同一个 **文件表项**
  - 文件偏移量是共享的

👉 这就是为什么：

- 父子进程同时写文件会交错
- 需要加锁 / 协调

------

## 五、那如果“想共享数据”，怎么办？

> **进程间通信（IPC）**

父子进程（或任意进程）要共享数据，必须**显式使用 IPC 机制**。

------

### 常见 IPC 方式一览

| 方式             | 是否共享内存 | 特点       |
| ---------------- | ------------ | ---------- |
| 管道（pipe）     | ❌            | 单向、简单 |
| 命名管道（FIFO） | ❌            | 有文件名   |
| 消息队列         | ❌            | 内核缓冲   |
| 信号             | ❌            | 控制通知   |
| **共享内存**     | ✅            | 真正共享   |
| Socket           | ❌            | 网络/本地  |

------

### 🔥 共享内存（真正的数据共享）

```
父进程 mmap/shmget
      ↓
子进程继承映射
      ↓
读写同一块物理内存
```

- 这是：
  - **最快**
  - **最危险**
- 必须配合：
  - 信号量 / 互斥锁
  - 否则会数据竞争

------

## 六、父子进程的“生命周期关系”

### 1️⃣ 子进程结束，父进程没 wait

- 子进程变成 **僵尸进程（Zombie）**
- 只剩 PID 和退出状态

### 2️⃣ 父进程先退出

- 子进程会被 **init / systemd 收养**
- 父 PID 变为 1

------

## 七、进程 vs 线程（顺便对比一下）

这是很多人混淆的点。

| 对比     | 进程       | 线程     |
| -------- | ---------- | -------- |
| 地址空间 | 独立       | 共享     |
| 数据共享 | 默认不共享 | 默认共享 |
| 崩溃影响 | 互不影响   | 可能全挂 |
| 创建开销 | 大         | 小       |
| 通信成本 | 高         | 低       |

👉 **线程是“真共享内存”，进程是“强隔离”**

------

## 八、一句“面试级总结”

你可以这样回答：

> 父进程是创建子进程的进程，子进程通常通过 fork 产生。
>  fork 后父子进程在逻辑上相关，但在内存上相互隔离，
>  普通变量不会共享，修改会触发写时复制。
>  如果需要共享数据，必须通过 IPC 机制（如共享内存、管道、socket）显式实现。

------

## 九、常见误区（一定要避开）

❌ 认为 fork 后变量天然共享
 ❌ 认为地址相同就一定是同一内存
 ❌ 把进程当线程用
 ❌ 忽略文件描述符共享带来的问题
