这是操作系统中非常核心、也非常容易被误解的概念。下面按“定义 -> 创建方式 -> 数据关系 -> 真正的共享方式 -> 常见误区”的顺序，把父子进程讲清楚。

## 一、什么是父进程 / 子进程？

### 1. 定义（操作系统视角）

- 父进程（Parent Process）：创建其他进程的进程。
- 子进程（Child Process）：由父进程创建出来的新进程。

在类 Unix 系统（Linux / macOS）中：

- 几乎所有进程（除 `init/systemd` 外）都有父进程。
- 所有进程会形成一棵进程树。

### 2. 父子进程是如何产生的？

在 Linux 中，最经典的方式是调用 `fork()`：

```c
pid_t pid = fork();
```

- `fork()` 只调用一次，但会返回两次。
- 在父进程中，返回值是子进程的 PID。
- 在子进程中，返回值是 `0`。

可以把这个过程理解为：

1. 父进程执行 `fork()`。
2. 内核创建出一个新的子进程。
3. 父子进程从 `fork()` 返回点继续往下执行。

## 二、最重要的结论

> 父子进程在逻辑上有关联，但在内存上彼此隔离。

这是理解后面所有问题的核心。

## 三、父子进程会共享数据吗？

### 1. 结论先说

> 父子进程默认不共享普通内存数据。

### 2. `fork()` 之后到底发生了什么？

当父进程调用 `fork()` 创建子进程时：

- 子进程会得到父进程虚拟地址空间的一份副本。
- 代码段、数据段、堆、栈等映射关系会被复制过去。
- 这里复制的是“进程视角下的地址空间”，并不是直接把两边变成同一块可写物理内存。

### 3. Copy-On-Write（写时复制）机制

为了避免一上来就完整复制所有内存页，操作系统通常会使用 COW（写时复制）：

- `fork()` 完成后，父子进程最开始可以暂时共享同一批物理页，但这些页会被标记为只读。
- 当任意一方尝试写内存时，操作系统才会真正复制对应的物理页。
- 从那一刻开始，父子进程修改的是各自独立的内存页，互不影响。

所以可以把它概括成一句话：

- 读的时候看起来像共享。
- 一旦写入，就立即隔离。

### 4. 一个直观例子

```c
int x = 10;
pid_t pid = fork();

if (pid == 0) {
    x = 20;
    printf("child: %d\n", x);
} else {
    sleep(1);
    printf("parent: %d\n", x);
}
```

输出通常会是：

```text
child: 20
parent: 10
```

原因很简单：子进程对 `x` 的写操作触发了写时复制，父进程中的那份 `x` 仍然保持原值。

## 四、那父子进程到底“共享”什么？

虽然父子进程不共享普通变量，但它们会继承一些内核资源的引用关系。

| 资源         | 是否共享 | 说明               |
| ------------ | -------- | ------------------ |
| 代码段       | 逻辑相同 | 程序代码一致       |
| 虚拟地址布局 | 相同     | 地址值看起来一样   |
| 文件描述符   | 共享     | 指向同一个打开文件 |
| 当前工作目录 | 继承     | `fork` 时复制      |
| 环境变量     | 复制     | 之后互不影响       |
| 信号处理方式 | 继承     | 可独立修改         |

### 文件描述符共享是最容易踩坑的地方

```c
int fd = open("a.txt", O_WRONLY);
fork();
```

此时父子进程都会持有 `fd`，而且它们指向的是同一个打开文件表项。这意味着：

- 文件偏移量可能是共享的。
- 父子进程同时写文件时，内容可能交错。
- 如果没有额外同步，就很容易出现顺序混乱的问题。

## 五、如果真的想共享数据，应该怎么办？

答案不是“指望 `fork()` 自动共享”，而是显式使用 IPC（进程间通信）机制。

常见方式如下：

| 方式             | 是否共享内存 | 特点       |
| ---------------- | ------------ | ---------- |
| 管道（pipe）     | 否           | 单向、简单 |
| 命名管道（FIFO） | 否           | 有文件名   |
| 消息队列         | 否           | 内核缓冲   |
| 信号             | 否           | 控制通知   |
| 共享内存         | 是           | 真正共享   |
| Socket           | 否           | 网络/本地  |

### 共享内存才是真正的数据共享

共享内存的过程可以概括成三步：

1. 父进程通过 `mmap` 或 `shmget` 创建共享区域。
2. 子进程继承这段映射。
3. 父子进程读写同一块物理内存。

它的优点是速度快，但风险也最高，因为：

- 多个执行流会同时访问同一份数据。
- 如果不配合信号量、互斥锁等同步手段，就会出现数据竞争。

## 六、父子进程的生命周期关系

### 1. 子进程结束，父进程没有 `wait`

- 子进程会变成僵尸进程（Zombie）。
- 此时它虽然已经退出，但内核还会保留 PID 和退出状态，等待父进程回收。

### 2. 父进程先退出

- 子进程会被 `init` 或 `systemd` 收养。
- 它的父 PID 会变成 `1`。

## 七、进程和线程有什么区别？

这也是最容易混淆的一组概念。

| 对比     | 进程       | 线程     |
| -------- | ---------- | -------- |
| 地址空间 | 独立       | 共享     |
| 数据共享 | 默认不共享 | 默认共享 |
| 崩溃影响 | 互不影响   | 可能全挂 |
| 创建开销 | 大         | 小       |
| 通信成本 | 高         | 低       |

一句话概括就是：线程是真共享内存，进程是强隔离。

## 八、一句话总结

> 父进程是创建子进程的进程，子进程通常通过 `fork()` 产生。`fork()` 之后，父子进程在逻辑上相关，但在内存上相互隔离；普通变量不会共享，写操作会触发写时复制。如果确实需要共享数据，就必须显式使用 IPC 机制，例如共享内存、管道或 Socket。

## 九、常见误区

- 认为 `fork()` 之后变量会天然共享。
- 认为地址值相同就一定指向同一块物理内存。
- 把进程当成线程来理解。
- 忽略文件描述符共享带来的副作用。
