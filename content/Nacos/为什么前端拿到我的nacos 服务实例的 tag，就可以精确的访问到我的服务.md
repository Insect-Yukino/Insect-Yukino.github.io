Nacos 的 Tag 是什么

```text
service: order-service
instances:
  - ip: 10.0.0.1  tag=gray
  - ip: 10.0.0.2  tag=gray
  - ip: 10.0.0.3  tag=prod
```

**tag 的本质是：实例元数据（metadata）的一部分**。

- 用于：
  - 灰度发布
  - 多环境路由
  - 定向流量
- 不用于：
  - 安全鉴权
  - 服务暴露

## 为什么前端“拿到 tag 就能精确访问”？（核心原理）

### 关键点只有一句话：

> **负载均衡器在“选实例”时，把 tag 当作筛选条件**

真实的调用链路
```text
1️⃣ 前端
   └─ 请求中携带 tag（Header / 参数）

2️⃣ 网关 / 后端
   └─ 从请求中读取 tag

3️⃣ 负载均衡器
   └─ 向 Nacos 拉服务实例
   └─ 只保留 tag = gray 的实例

4️⃣ 最终请求
   └─ 精确打到 gray 服务实例
```

前端请求
```http
GET /api/order/list
X-Env-Tag: gray
```

网关读取

```java
String tag = request.getHeader("X-Env-Tag");
```

#### 负载均衡器筛选实例（伪代码）

```
instances.stream()
    .filter(i -> tag.equals(i.getMetadata().get("tag")))
    .collect(...)
```

> ⚠️ 这一步 **100% 在后端完成**

------

## 四、为什么“看起来像前端精确访问了服务”？

### 因为：

- tag 是 **确定性的**
- 路由规则是 **硬过滤**
- 不再是「随机 / 轮询」

所以你会感觉：

> “我只要给这个 tag，就一定能访问到那台服务”

但本质是：

> **前端只是参与了“路由条件的输入”，不是服务发现**

------

## 五、那为什么官方和架构上不推荐前端参与？

### 🚨 原因 1：tag ≠ 安全边界

```
tag = prod
tag = gray
tag = internal
```

前端可以随便改：

```
X-Env-Tag: internal
```

👉 **等于绕过发布控制**

------

### 🚨 原因 2：破坏流量治理

- 灰度比例
- 权重控制
- 动态调整

都会失效，因为前端是「全量命中」。

------

### 🚨 原因 3：耦合服务部署细节

- 前端不该知道：
  - 有多少实例
  - 有多少环境
  - 哪些是灰度

------

## 六、那 tag 正确该由谁决定？

### ✅ 推荐做法（企业级）

| 决策来源   | 示例         |
| ---------- | ------------ |
| 用户维度   | 用户 ID Hash |
| Token 信息 | JWT claim    |
| 登录态     | 内部账号     |
| IP / UA    | 内部访问     |
| 配置中心   | 灰度比例     |

> **由网关或后端计算 tag，而不是前端指定**

------

### 示例：网关自动打 tag

```
if (user.isGrayUser()) {
    request.addHeader("X-Env-Tag", "gray");
}
```

前端完全无感。

------

## 七、一个“你现在这种情况”的真实判断

你现在的现象说明：

> ✔ 你已经 **开启了 Nacos 标签路由**
>  ✔ 网关或 LoadBalancer **启用了基于 metadata 的实例过滤**
>  ✔ 前端 tag 被 **当成了路由条件**

这在**技术上是可行的**，但在**架构上不推荐长期使用**。

------

## 八、一句话总结（适合写在设计说明里）

> 前端携带的 tag 并不会直接访问 Nacos 或选择服务实例，而是作为请求条件传递给后端，由后端在进行服务发现和负载均衡时，根据该 tag 对 Nacos 返回的服务实例进行过滤，从而实现定向路由或灰度发布。真正的实例选择与负载均衡逻辑始终发生在后端或网关层。
