如何优雅的通过 FeignClient 调用另一个微服务的方法

下面给你一个「**优雅**」的 Feign 调用案例：**consumer（system-service）** 调用 **provider（infra-service）** 的用户查询接口。重点放在：**API 解耦、统一返回、传递鉴权/链路ID、超时/重试、统一异常、fallback（可选）**。

------

## 1）推荐结构：拆一个 `xxx-api` 模块做契约（最优雅）

把 **DTO + FeignClient 接口** 放到独立模块，比如：`iwater-module-infra-api`，consumer 和 provider 都依赖它。

好处：

- consumer 不用“手写一套 DTO/路径”，避免接口漂移
- provider 改动能编译期暴露问题

------

## 2）Provider（被调用方）示例：infra-service

```
@RestController
@RequestMapping("/admin-api/infra/user")
public class InfraUserController {

    @GetMapping("/{id}")
    public CommonResult<UserRespDTO> getUser(@PathVariable Long id) {
        UserRespDTO dto = new UserRespDTO();
        dto.setId(id);
        dto.setUsername("tom");
        return CommonResult.success(dto);
    }
}
```

------

## 3）API 模块：FeignClient + DTO（契约）

```
@FeignClient(
        name = "iwater-module-infra",              // 注册中心的服务名
        path = "/admin-api/infra/user",            // 统一前缀
        contextId = "infraUserApi"
        // fallbackFactory = InfraUserApiFallbackFactory.class  // 可选：容错
)
public interface InfraUserApi {

    @GetMapping("/{id}")
    CommonResult<UserRespDTO> getUser(@PathVariable("id") Long id);
}
```

DTO 示例：

```
@Data
public class UserRespDTO {
    private Long id;
    private String username;
}
```

------

## 4）Consumer（调用方）示例：system-service

### 4.1 启用 Feign

```
@SpringBootApplication
@EnableFeignClients(basePackages = "cn.iwater.module.infra.api") // 指向 API 模块包
public class SystemApplication {}
```

### 4.2 业务里调用（建议做一层 Facade/Service，别在 Controller 直接 call）

```
@Service
@RequiredArgsConstructor
public class UserFacade {

    private final InfraUserApi infraUserApi;

    public UserRespDTO getUserOrThrow(Long id) {
        CommonResult<UserRespDTO> res = infraUserApi.getUser(id);
        if (res == null || !res.isSuccess()) {
            throw new RuntimeException("调用 infra 用户接口失败：" + (res == null ? "null" : res.getMsg()));
        }
        return res.getData();
    }
}
```

Controller：

```
@RestController
@RequiredArgsConstructor
@RequestMapping("/admin-api/system/demo")
public class DemoController {

    private final UserFacade userFacade;

    @GetMapping("/user/{id}")
    public CommonResult<UserRespDTO> demo(@PathVariable Long id) {
        return CommonResult.success(userFacade.getUserOrThrow(id));
    }
}
```

## 5）“优雅”的关键配套：拦截器透传鉴权 + 链路ID

很多微服务会需要把当前请求的 token、traceId 传给下游。

```
@Configuration
public class FeignCommonConfig {

    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            // 1) 透传 Authorization（示例：从 SpringMVC 请求头取）
            ServletRequestAttributes attrs =
                    (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
            if (attrs != null) {
                String auth = attrs.getRequest().getHeader("Authorization");
                if (auth != null) template.header("Authorization", auth);

                String traceId = attrs.getRequest().getHeader("X-Trace-Id");
                if (traceId != null) template.header("X-Trace-Id", traceId);
            }
        };
    }
}
```

然后在 `@FeignClient(configuration = FeignCommonConfig.class)` 或全局生效（放到 `@Configuration` 被扫描即可）。

## 6）超时/日志（必须配）

```
spring:
  cloud:
    openfeign:
      client:
        config:
          default:
            connectTimeout: 2000
            readTimeout: 5000
            loggerLevel: basic
```

日志级别（可选）：

```
@Bean
public feign.Logger.Level feignLoggerLevel() {
    return feign.Logger.Level.BASIC; // 生产建议 BASIC，排障再 FULL
}
```

------

## 7）统一把非 2xx 变成可控异常（更“优雅”）

默认 Feign 遇到 4xx/5xx 会抛 FeignException；你可以用 `ErrorDecoder` 统一转业务异常：

```
@Bean
public ErrorDecoder errorDecoder() {
    return (methodKey, response) -> {
        return new RuntimeException("Feign调用失败: " + methodKey + ", http=" + response.status());
    };
}
```

------

## 8）可选：fallback（服务降级）

如果你用 Resilience4j/Spring Cloud CircuitBreaker，可配 `fallbackFactory`：

```
@Component
public class InfraUserApiFallbackFactory implements FallbackFactory<InfraUserApi> {
    @Override
    public InfraUserApi create(Throwable cause) {
        return id -> CommonResult.error(500, "infra user 服务不可用：" + cause.getMessage());
    }
}
```

# 你按这个模式落地，基本就“优雅”了

**总结最佳实践**（你项目很适配）：

- ✅ 接口契约下沉到 `xxx-api` 模块（DTO + FeignClient）
- ✅ consumer 做 Facade/Service 层封装，不要 Controller 直调
- ✅ 透传 token / traceId
- ✅ 配超时（必做）+ 统一错误处理（强烈建议）
- ✅ 需要时加 fallback / 熔断

你看到的这种写法：

```
@RestController
@Validated
public class PermissionApiImpl implements PermissionApi
```

它的目的通常不是“为了让 Feign 能调用”（Feign 只需要 HTTP 接口即可），而是为了**把同一份接口契约（PermissionApi）同时用于：**

1. **对外暴露 HTTP 接口（Provider 端）**
    `PermissionApiImpl` 是服务提供方真正的 Controller，实现 `PermissionApi` 里定义的路径、入参、返回值。
2. **给 FeignClient 复用同一份方法签名（Consumer 端）**
    Consumer 端会写：

```
@FeignClient(name="xxx", path="/xxx")
public interface PermissionApi { ... }
```

这样 Provider 和 Consumer **共享同一个接口定义**，避免：

- URL 写错
- 参数名/PathVariable 名不一致
- DTO 重复维护
- 接口改了 consumer 不知道（编译期就能发现）

换句话说：**PermissionApi 是契约，PermissionApiImpl 是契约的 HTTP 实现。**

------

### 这套模式带来的实际好处

- ✅ **接口强一致**：改接口会编译报错，逼着两端一起改（比接口文档靠谱）
- ✅ **减少重复代码**：DTO、返回值、注解（@GetMapping/@PostMapping 等）都只写一份
- ✅ **降低联调成本**：Feign 调用跟 Controller 保持同签名，少踩坑
- ✅ **自动参数校验**：`@Validated` 让接口入参（@NotNull/@Valid）在 provider 端统一生效

------

### 需要注意的坑（你项目里很可能会遇到）

1. **不要把业务逻辑写在 Impl 里**
    Impl 只做参数接收/校验/调用 Service，业务逻辑放 Service，否则会很难测、也不好复用。
2. **接口层不要耦合“内部对象”**
    契约模块里尽量只放 DTO/VO，不要暴露 DO/Entity。
3. **接口注解放哪？**

- 常见做法：**Mapping 注解放在接口 PermissionApi 上**，Impl 只实现方法即可
- 或者：注解放在 Impl 上（接口只留方法签名）
   第一种更利于“契约即文档”，也更利于 consumer 复用（很多团队推荐）。
