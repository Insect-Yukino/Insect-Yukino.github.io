一致性哈希的虚拟节点是怎么添加的

一致性哈希里“加虚拟节点”的本质：**同一台物理节点在哈希环上放多个位置**，用这些位置把环切成更多小区间，从而让数据分布更均匀、扩缩容时迁移更平滑。

------

## 1）虚拟节点怎么“添加”到哈希环

假设物理节点是 `nodeA`，你给它配置 `V` 个虚拟节点（replicas / vnodes）：

- 生成虚拟节点标识：
   `nodeA#0, nodeA#1, ..., nodeA#(V-1)`
- 对每个虚拟节点标识做 hash，得到一个环上的位置（0 ~ 2^32-1 或 0~2^64-1）
- 把这些位置插入“环”（有序结构）里，并在位置上标记它属于 `nodeA`

这样一台机器就不是一个点，而是 `V` 个点散布在环上。

> `V` 越大，分布越均匀，但路由表越大、内存/维护成本越高。常见几十到几百个 vnode/节点（看规模）。

------

## 2）查询 key 落到哪个物理节点（含 vnode）

对业务 key（比如 userId）：

1. `h = hash(key)`
2. 在环上找**第一个位置 >= h** 的虚拟节点（顺时针）
3. 该虚拟节点映射到它的物理节点，就是目标机器
4. 如果没找到（h 比最大点还大），就绕回环头（第一个点）

------

## 3）为什么虚拟节点能让数据更均匀

如果每台机器只有 1 个点：

- 点的位置随机，区间大小差异可能很大 → 有的机器扛很多 key，有的很少

每台机器放很多 vnode：

- 每个机器拿到很多小区间的集合
- 大区间被拆碎并平均分配 → 总体更均匀（大数定律）

------

## 4）扩容/缩容时怎么做（虚拟节点视角）

### 扩容：新增物理节点 nodeX

- 给 nodeX 创建 `V` 个 vnode，插入环
- 只有这些 vnode “插入点” 所覆盖到的顺时针区间会迁移到 nodeX
- 其它节点数据不动（迁移量远小于取模）

### 缩容：下线 nodeA

- 把 nodeA 的所有 vnode 从环上移除
- 这些 vnode 的区间数据会顺时针迁移到它们的后继 vnode 所属节点

------

## 5）工程实现常用数据结构（Java 思路）

用一个有序 map 维护环（key=hash位置，value=物理节点ID）：

- Java 常用：`TreeMap<Long, Node>`
- 查找用：`ceilingEntry(h)` 找顺时针第一个点
- 没有就用 `firstEntry()` 回环头

虚拟节点插入就是循环 `replica` 次 `put(hash(nodeId+"#i"), node)`。

------

## 6）注意点（真实线上会踩）

1. **hash 函数要好**：用 MurmurHash/xxHash 这类分布均匀的，别用很弱的 hash。
2. **vnode 数怎么选**：
   - 节点少（<10）时 vnode 建议多一点（如 200~1000）更均匀
   - 节点多时 vnode 可少一些（如 50~200），路由表别太大
3. **权重节点（异构机器）**：大机器配置更多 vnode，小机器少 vnode，按权重分配容量。
4. **元数据一致**：所有服务实例必须用同一份“环”（放配置中心/注册中心），否则路由会不一致。
5. **迁移和一致性**：一致性哈希只解决“映射变化小”，不自动帮你迁数据；还需要搬迁流程/双写/对账。
