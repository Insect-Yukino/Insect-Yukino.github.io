## 它主要解决什么问题

它最适合解决这类分布式事务场景：

**本地数据库要提交，同时还要把一条消息发到别的系统**。
 比如：

- 订单创建成功，同时通知库存系统扣减
- 用户注册成功，同时发欢迎短信/邮件
- 支付成功，同时通知账单、积分、风控
- 一个服务更新本地聚合后，还要发布领域事件给其他服务

这类问题的本质是 **双写问题**：一次业务操作里要同时写数据库和消息系统。只要其中一个成功、另一个失败，就会产生不一致。Transactional Outbox 的目的就是避免这种“双写不一致”。

## 它能应对哪些分布式事务场景

### 1. 服务内本地事务 + 跨服务异步通知

这是最典型的场景。
 你的服务只对自己的数据库做本地事务，但事务提交后需要把变更可靠地通知出去。Outbox 会把“业务数据”和“待发送消息”放在**同一个本地事务**里提交，这样两者要么一起成功，要么一起失败。

### 2. Saga 的步骤推进

Saga 的每一步本质上都是：

- 更新本地状态
- 发一条事件给下一个参与方

Outbox 很适合用在 Saga 的每个本地事务里，保证“本地状态变了，对外事件一定最终可见”。microservices.io 明确把 Saga 和 Transactional Outbox 关联起来使用。

### 3. 领域事件发布

例如订单状态从 `CREATED` 变为 `PAID`，你想发布 `OrderPaid` 事件给搜索、推荐、报表等服务。Outbox 能保证事件和状态变更的一致性，避免“数据库已更新，但事件没发出去”。

### 4. 最终一致性场景

它适合的是 **最终一致性**，不是强一致分布式提交。也就是说，消费者会稍后收到消息，但不会要求所有系统在一个全局事务里同时提交。AWS 和 Debezium 的资料都把它定位为可靠异步数据交换方案。

## 它不适合什么场景

### 1. 你要求强一致、立即全局提交

Outbox 不是 2PC。它解决的是“最终一定会发出去”，不是“所有系统同时原子提交”。如果你要求扣款成功的瞬间库存、账务、风控全部立刻同事务成功，这不是 Outbox 的目标。

### 2. 需要跨多个数据库做同步原子写

Outbox 只能保证“本服务自己的数据库状态”和“本服务要发出的消息”一致。它不能替你把 A 库、B 库、C 库做成一个数据库级原子事务。跨服务数据一致仍然要靠 Saga、补偿、重试来完成。

## 你该怎么处理

最实用的做法是按下面这套来。

### 第一步：建一张 outbox 表

常见字段：

- `id`
- `aggregate_type`
- `aggregate_id`
- `event_type`
- `payload`
- `headers`
- `status`
- `created_at`
- `published_at`
- `retry_count`

如果你走 Debezium CDC，通常会有一个专门的 outbox 表，连接器从这张表捕获变更并路由到消息系统。Debezium 官方就是这么设计的。

### 第二步：业务表变更和 outbox 插入放在同一个本地事务里

例如：

```
@Transactional
public void createOrder(...) {
    orderMapper.insert(order);      // 业务数据
    outboxMapper.insert(eventRow);  // 待发送消息
}
```

这样只要事务成功提交，订单和消息记录一定同时存在；事务回滚则两者都没有。这个就是 Transactional Outbox 的核心。

### 第三步：把 outbox 表里的消息发出去

有两种主流办法。

#### 方案 A：轮询发布（Polling Publisher）

应用自己定时扫 `outbox` 表，把未发送消息发到 Kafka/RabbitMQ，然后更新状态。microservices.io 把它列为 Outbox 的一个常见配套模式。

适合：

- 先快速落地
- 技术栈简单
- 没有 CDC 基础设施

#### 方案 B：CDC / 事务日志订阅

用 Debezium 之类去读数据库日志，把 outbox 表变化自动推送到消息系统。Debezium 官方的 Outbox Event Router 就是专门干这个的。

适合：

- 可靠性要求更高
- 不想自己写轮询发布器
- 已有 Kafka Connect / Debezium 体系

### 第四步：消费者必须幂等

Outbox 只能保证“至少一次”投递更容易做到；如果发布端重试，消费者就可能收到重复消息。所以消费者要按业务键或事件 ID 做去重。AWS 明确提醒消费者要有幂等性。

常见做法：

- 消费端建一张 `processed_event` 表，按 `event_id` 去重
- 或在目标表上做唯一约束
- 或业务逻辑天然幂等

### 第五步：设计重试和死信

你需要考虑：

- 发布失败怎么重试
- 重试多少次
- 长时间失败怎么人工介入
- 是否要死信队列

否则 outbox 表会越积越多，或者消息永远卡住。

### 第六步：清理历史 outbox 数据

消息成功发布并确认后，要做归档或删除策略，不然表会膨胀，轮询和 CDC 成本都会变高。

## 一个典型落地流程

以“创建订单后通知库存”为例：

1. 订单服务开启本地事务
2. 写 `orders` 表
3. 同事务插入一条 `outbox`：`OrderCreated`
4. 事务提交
5. 发布器或 Debezium 把 `OrderCreated` 发到 MQ / Kafka
6. 库存服务消费事件并扣库存
7. 如果库存失败，走 Saga 补偿或重试

这样你避免了最危险的情况：

- 订单写成功，但消息没发
- 消息发了，但订单没提交

因为订单和 outbox 已经绑在一个本地事务里了。

## 什么时候该选 Outbox

适合选它的信号通常是：

- 你有“数据库更新 + 发消息”的双写
- 你接受最终一致性
- 你不想上 2PC
- 你在做微服务事件驱动
- 你在做 Saga

## 你现在可以直接照着做

如果你是 Spring + MySQL / Kingbase 这一类项目，最稳的路线通常是：

- 先建 `outbox_event` 表
- 在业务 `@Transactional` 里同时写业务表和 `outbox_event`
- 先用定时轮询发布，快速跑通
- 后面再升级成 Debezium CDC
- 消费端强制幂等
