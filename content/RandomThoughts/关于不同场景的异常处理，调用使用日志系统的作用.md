## 一、先把异常处理拆成 3 个维度

### 1）异常该不该往上抛

- **必须抛**：失败会导致业务结果不可信/不可用（核心链路、强一致、权限、扣费、核心搜索等）
- **可以吞但要降级**：失败不影响主流程，允许退化（推荐位、统计、非核心补充信息）
- **不抛也不吞**：用“返回值模型”表达失败（例如 `Result<T>` / `Either` / `Optional`），让调用方显式分支处理

### 2）用户/调用方要不要感知

- **需要明确感知**：展示错误提示、返回错误码（如 503），避免把故障伪装成“无数据”
- **不需要强感知**：静默降级（比如“猜你喜欢”挂了就不展示）

### 3）系统要不要“可观测”

- **永远需要**：对外部依赖失败必须可观测（日志 + 指标 + trace），否则线上等于“瞎子摸象”

## 二、常见场景的最佳实践（按业务影响分级）

### 场景 A：核心链路（必须正确，失败就是失败）

**例子**：ES 核心搜索、支付扣费、鉴权、下单、生成关键报表
 **做法**：

1. **记录 error 日志**（带关键上下文 + 异常堆栈）
2. **向上抛出异常**（包装成业务异常/外部依赖异常）
3. **统一异常处理**返回明确错误码/错误信息（如 503）
4. 可选：**有限重试**（只对短暂可恢复错误：IO/超时/429/503），失败仍然抛

**不要做**：catch 后 `return emptyList()` —— 会把“系统故障”伪装成“无结果”。

------

### 场景 B：可降级链路（允许退化，但服务要稳）

**例子**：推荐列表、相似内容、热度统计、非关键标签信息
 **做法**：

1. **warn 日志**（说明“已降级”，不要每次都打满堆栈）
2. **返回降级结果**：缓存/默认值/空列表（必须明确这是降级）
3. **打指标**：失败率、降级次数（便于告警）

**关键点**：用户可不感知，但运维/开发必须感知（否则长期“悄悄坏掉”）。

------

### 场景 C：参数/业务校验失败（可预期错误）

**例子**：参数为空、格式错误、业务规则不满足
 **做法**：

- 直接返回 **4xx 或业务错误码**
- 通常不需要 error 堆栈（因为不是系统异常）
- 可选：`info/warn` 记录“关键入参 + 原因”，用于风控/排查

**不要做**：把它当成系统异常打 error 堆栈（会污染告警）。

------

### 场景 D：编程错误/系统 bug（不可预期）

**例子**：NPE、数组越界、序列化 bug
 **做法**：

- **必须 error + 堆栈**
- 让它暴露出来（抛给全局异常处理），尽快修复
- 这类问题通常不该靠重试解决

------

### 场景 E：资源/线程/关闭失败（边界层异常）

**例子**：关闭流失败、线程池 shutdown 异常
 **做法**：

- 大多数情况下：`warn` + 记录异常，不要影响主流程
- 如果会造成数据损坏/资源泄露不可控：升级为 `error` 并触发告警

------

## 三、日志系统的作用：它不只是“打印到控制台”

### 1）持久化与集中化（排障的“证据链”）

- 本地开发：你看到的是控制台
- 线上生产：日志往往会被采集到文件/日志平台（ELK、Loki、Splunk、云日志）
- 没日志就没证据：问题发生当下的上下文（参数、traceId、节点、堆栈）会消失

### 2）检索与关联（能定位“哪一次请求”）

合理的日志字段可以让你：

- 按 **traceId/requestId** 串起一次请求全链路
- 按 `index=content`、`status=503` 过滤聚类
- 按关键词、用户、接口定位高频失败路径

### 3）告警与运营指标的来源

日志/指标是告警基础：

- `error` 激增、某类异常突然变多 → 自动告警
- 降级次数/失败率 > 阈值 → 提醒处理

### 4）分级控制与成本控制

日志有 level（trace/debug/info/warn/error）：

- **error**：系统异常、需要处理（通常会告警）
- **warn**：可恢复/已降级/值得关注
- **info**：关键业务事件（下单成功、状态变更）
- **debug/trace**：开发期排查（线上通常关闭）

用日志系统你还能做：

- 异步写日志、滚动文件、保留天数、单文件大小限制
- 不同模块不同级别（避免把磁盘打满）

------

## 四、怎么判断“该不该打日志、打什么级别”（快速口诀）

### 1）是否需要人来处理？

- 需要立刻处理/影响用户：**error**
- 不影响主流程但值得关注：**warn**
- 正常业务过程记录：**info**
- 开发排查细节：**debug/trace**

### 2）是否可预期？

- 预期内的用户错误：一般 **不打 error**
- 不可预期的系统异常：**error + 堆栈**

### 3）是否会“吞掉异常/降级”？

- 如果你吞掉了异常：**必须至少 warn，并打指标**
   否则系统会“悄悄坏”。

------

## 五、落到你当前场景：ES 核心搜索怎么做最合理

- **IO/超时/429/503**：可尝试 1~2 次重试；仍失败 → **error 日志 + 抛异常** → 返回明确错误（503/业务码），前端提示“搜索暂不可用”
- 不要把异常变成空列表（会误导用户为“没有搜到”）
