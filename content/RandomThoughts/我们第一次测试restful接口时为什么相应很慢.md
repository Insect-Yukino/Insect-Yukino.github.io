这里我们就说最简单的查询数据库的接口。

### A. JVM & Spring Boot 冷启动

第一次打到接口时，可能会触发：

- **类加载 / JIT 预热**：某些热点方法第一次执行会慢，后面 JIT 编译优化后会快
- **Spring 容器懒加载 Bean**：如果开启了懒加载，第一次触发某个 Bean 才会实例化、注入依赖、AOP 代理等
- **AOP / 拦截器链初始化**：比如事务 `@Transactional`、权限、日志、审计等的代理对象/增强链

> 很多项目并不是默认懒加载，Spring Boot 默认是 **非懒加载**，但大家可能手动开过 `spring.main.lazy-initialization=true` 或某些 Bean 因条件装配/延迟依赖导致看起来像懒加载。

### B. MyBatis / MyBatis-Plus 首次使用成本

第一次调用 mapper 时，可能发生：

- **Mapper 代理创建**
- **解析/加载 SQL 映射**：XML 或注解 SQL、动态 SQL、resultMap、typeHandler 等
- **反射元数据缓存建立**：MyBatis 会缓存反射信息、映射信息，后续会快很多

### C. 数据库连接池第一次借连接成本

第一次访问 DB 还可能慢在：

- **连接池创建/初始化**
- **首次建立物理连接**：TCP 握手、认证、SSL、server variables 拉取等
- **首次校验连接**
- **PreparedStatement 缓存未命中**

> 很多时候，真正拉胯的是首次建立连接而不是 SQL 本身。

### D. 数据库端的冷缓存 / 计划缓存 / IO

- **Buffer Pool / Page Cache 冷**：数据页不在内存，需要磁盘读
- **索引页冷**：第一次走索引也要读索引页
- **执行计划缓存**

> 索引和数据在数据库层是以页的形式存储在文件的，开始数据库的缓存中并没有索引页和数据页，但是等进行一些操作之后，索引页和数据就会被缓存在 Buffer Pool 中，这样再进行操作时会直接操作缓存，速度很快。

### E. 其他一次性初始化

- **日志框架初始化**（Logback 异步队列、Appender 打开文件等）
- **序列化组件首次使用**（Jackson 创建 ObjectMapper、反射缓存）
- **DNS 解析**（调用别的服务/DB 域名，首次解析慢）
- **安全框架**（Spring Security、JWT 公钥加载）
- **配置中心/注册中心**（Nacos、Apollo 首次拉配置/心跳）

综上，当我们访问一次接口过后，Spring Bean 等一些资源肯定加载完毕了，下一次访问时直接命中 MySQL 的缓存，这样也极大的降低响应时间，这就是为什么第一次访问接口会明显的比之后访问接口要慢。

> 如果公司是微服务项目，不同服务之间的调用也很可能会影响当前接口的响应时间，特别是别的服务网络或者其他方面出问题时。。。
