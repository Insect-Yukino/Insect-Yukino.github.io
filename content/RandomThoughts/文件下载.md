如何在服务器上部署可下载文件？

在 Linux 服务器上暴露一个文件的下载地址，最安全的实践通常不是把目录直接公开，而是将 **文件放在不对外公开的位置（私有目录） → 由应用做鉴权/授权 → 让 Nginx 负责真正传文件（高性能）**

经典方案：**Nginx `X-Accel-Redirect`**

下面给你一套生产里很常用、也很安全的落地方式。

### 方案一：应用鉴权 + Nginx 内部下载（X-Accel-Redirect）

文件不要放在 web 根目录

比如放到：

- `/data/downloads/private/`
- 权限只给运行服务的用户（或同组）读

```bash
sudo mkdir -p /data/downloads/private
sudo chown -R appuser:appgroup /data/downloads/private
sudo chmod 750 /data/downloads/private
```

Nginx 配置一个内部 location

Nginx **不会允许外部直接访问** 这个路径，只能由应用内部跳转触发下载：

```nginx
server {
  listen 443 ssl;
  server_name your.domain.com;

  # 外部下载入口（给应用接口用）
  # /api/download/xxx 由后端处理鉴权

  # 内部文件映射：只有 internal 跳转能进
  location /_protected_files/ {
    internal;
    alias /data/downloads/private/;

    # 建议：关闭目录列表
    autoindex off;

    # 建议：限速与防刷（按需）
    limit_rate 2m;
  }
}
```

解释：

- `internal;`：外部请求访问 `/_protected_files/...` 会被拒绝
- `alias`：把 URL 路径映射到真实磁盘目录

后端接口做权限校验，然后返回内部跳转头

以 Spring Boot 为例（核心是 `X-Accel-Redirect`）：

```java
@GetMapping("/api/download/{fileId}")
public ResponseEntity<Void> download(@PathVariable String fileId) {
    // 1) 鉴权：判断当前用户是否允许下载这个 fileId
    // 2) 查元数据：真实文件相对路径、原始文件名
    String relativePath = "2026/01/report.pdf"; // 举例：不要用用户输入直接拼路径
    String filename = "report.pdf";

    HttpHeaders headers = new HttpHeaders();
    headers.add("X-Accel-Redirect", "/_protected_files/" + relativePath);

    // 下载文件名（注意转义/编码）
    headers.add(HttpHeaders.CONTENT_DISPOSITION,
            "attachment; filename=\"" + filename.replace("\"","") + "\"");

    return ResponseEntity.ok().headers(headers).build();
}
```

这样：

- **权限在应用层控制**
- **传输由 Nginx 做（支持 sendfile、Range、性能好）**
- 真实路径不暴露给用户（URL 看不到 `/data/...`）

这套方案和我实习的公司使用的类似，只不过他是直接返回给前端服务器中可下载文件的地址，然后请求由 Nginx 转发到下载。上述聊的方案比较安全，原因是： 

1. **私有目录不对外公开**
2. **必须经过后端鉴权**
3. **防路径穿越**：不要把用户传来的 `../` 直接拼成路径
   - 最好用 **fileId → 数据库记录 → 受控相对路径**
4. **HTTPS**：下载链接和 Cookie/Token 不被窃听
5. **可加风控**：限速/限并发/限频/黑名单/审计日志
