**“耗性能”并不是因为‘切换’这两个字本身神秘，而是因为切换时要保存/恢复状态、改变执行环境、破坏 CPU 的局部性。**

你这个问题其实分成两层：

1. **用户态 ↔ 内核态切换** 为什么有开销
2. **线程/进程切换** 为什么有开销（以及线程和进程的区别）

另外有个容易混淆的点，我先说清楚：

> **普通的线程/进程切换，不可能纯发生在用户态。**
>  只要是 OS 线程或 OS 进程的调度切换，一定要经过内核调度器，所以一定包含一次“进入内核”。

真正能“只在用户态切换”的，是：

- 用户级线程（green thread）
- 协程
- 某些运行时自己做的调度

------

# 一、什么是用户态 / 内核态

可以简单理解为 CPU 的两种权限级别：

- **用户态**：权限低，应用程序运行在这里
- **内核态**：权限高，操作系统内核运行在这里

用户程序不能直接做这些敏感操作：

- 访问硬件设备
- 修改页表
- 执行特权指令
- 中断管理
- 进程调度

所以应用想做这些事时，只能通过：

- 系统调用（syscall）
- 异常（exception）
- 中断（interrupt）

让 CPU 从用户态进入内核态。

------

# 二、为什么“用户态 -> 内核态”切换会耗性能

注意：
 **单纯“权限位变一下”本身不是最贵的，真正贵的是伴随它发生的一整套动作。**

------

## 1）陷入内核需要“切换执行上下文”

比如你调用一次 `read()`、`write()`、`futex()`、`epoll_wait()`，CPU 不能直接继续按用户态那套上下文跑了，它要先进入内核的执行环境。

通常要做这些事：

### （1）保存当前用户态现场

要把当前线程在用户态的关键寄存器状态保存下来，例如：

- 程序计数器（PC / RIP）
- 栈指针（SP / RSP）
- 标志寄存器
- 通用寄存器

因为内核处理完之后，还得**准确返回到用户态原来那条指令附近继续执行**。

这一步本身就有成本。

------

### （2）切到内核栈

每个线程通常不只有用户栈，还有一份内核栈。
 进入内核后，CPU/内核需要切换到这个线程对应的**内核栈**来执行系统调用处理逻辑。

这相当于：

- 原来用用户栈
- 现在改用内核栈

虽然不是“很重”的大操作，但也是一次环境切换。

------

### （3）执行特权检查和入口分发

系统调用不是直接跳过去就完了，还要：

- 确认入口合法
- 根据 syscall 编号分发到对应内核函数
- 做参数检查（用户传来的指针是否合法、能不能访问）
- 处理访问边界

比如你传一个用户态地址给内核，内核不能盲信，往往要检查这块内存是否可读/可写。

这些都增加成本。

------

## 2）用户态和内核态之间的数据访问有额外成本

比如：

- `read(fd, buf, len)`
- `write(fd, buf, len)`

`buf` 是用户态地址，内核不能直接随便信任它。

通常要做：

- 检查地址是否合法
- 检查是否越界
- 检查页是否存在
- 必要时从用户态拷贝到内核态，或从内核态拷回用户态

所以很多系统调用慢，不是慢在“切权限”，而是慢在：

- **切进内核**
- **再做校验**
- **再做数据拷贝**
- **再返回**

------

## 3）CPU 流水线、分支预测、指令缓存的“局部性”会被打断

现代 CPU 为了快，有很多优化：

- 指令流水线
- 分支预测
- cache 局部性

当你从用户态突然陷入内核态时，本质上执行流跳到了另一套代码路径：

- 原来跑的是应用代码
- 现在跑的是内核代码

这会导致：

- 流水线被打断或重新填充
- 分支预测历史失效一部分
- 指令 cache / 数据 cache 的热点被污染

结果就是：
 **即使内核代码很短，也会破坏 CPU 刚建立好的“热身状态”。**

这在高频 syscall 场景下很明显。

------

## 4）安全机制会让切换更贵

现代 CPU/OS 为了防漏洞（比如 Meltdown/Spectre 类），在用户态和内核态切换时，往往加了更多防护。

这可能包括：

- 更严格的页表隔离
- 额外的屏障指令
- 减少特权信息泄漏的防护步骤

所以“进入内核再返回”的代价，在现代系统上通常比非常早期的系统更高。

------

## 5）如果系统调用会阻塞，还会引出更大的调度成本

最重要的一点：

> 很多系统调用不是“进去一下就回来”，而是会阻塞。

例如：

- 等磁盘
- 等网络
- 等锁
- 等事件

一旦阻塞，内核可能会：

1. 把当前线程挂起
2. 调度别的线程上 CPU

这时候成本就不只是“用户态↔内核态切换”了，
 而是升级成了**线程切换 / 进程切换**，开销更大。

------

# 三、线程切换为什么耗性能

先说结论：

**线程切换的成本，主要来自“保存/恢复上下文 + 调度器决策 + 破坏 cache/TLB 局部性”。**

------

## 1）保存当前线程的寄存器状态

线程 A 被换下去时，CPU 必须把它的执行现场存起来：

- 通用寄存器
- 程序计数器
- 栈指针
- 状态寄存器
- 可能还有浮点/SIMD 寄存器（这部分可能更贵）

否则以后切回 A 时，A 就没法从原位置继续跑。

------

## 2）恢复下一个线程的寄存器状态

线程 B 被换上来时，CPU 还要把 B 上次保存的现场恢复出来。

所以线程切换不是一个动作，而是一对动作：

- 保存 A
- 恢复 B

------

## 3）调度器本身需要工作

线程不是自动切的，而是内核调度器决定的。
 调度器要做的事包括：

- 看当前运行队列里谁能跑
- 判断优先级
- 处理时间片
- 处理唤醒/阻塞
- 可能涉及多核下的负载均衡

这些都是额外代码执行，不是“零成本”。

如果线程很多、竞争激烈，调度器本身也会有明显开销。

------

## 4）CPU cache 局部性被破坏

这是实际里非常大的成本来源。

线程 A 刚运行时，会把它常用的数据和代码“加热”到：

- L1/L2/L3 cache
- TLB
- 分支预测器状态

切到线程 B 后：

- B 用的是另一套数据
- 会把 A 的热点缓存挤掉一部分

等再切回 A 时，A 又得重新把自己的工作集热起来。

所以线程切换的代价，很多时候不是“保存几个寄存器”，而是：

**切换之后，cache 命中率变差，后续很多条指令都变慢。**

这叫“切换造成的间接成本”。

------

## 5）如果线程在不同 CPU 核间迁移，成本更高

如果线程不是在同一个核心上恢复，而是从 core 0 跑到 core 3：

- 原核心上的 cache 热数据利用不上了
- 新核心要重新加载

这样性能更差。

所以操作系统通常也会尽量考虑 **CPU 亲和性（affinity）**，减少乱迁移。

------

# 四、进程切换为什么比线程切换更贵

线程切换已经有成本了，
 **进程切换通常更贵**，因为除了线程上下文，还可能涉及“地址空间切换”。

------

## 1）进程有不同的虚拟地址空间

不同进程通常有不同的：

- 页表
- 虚拟内存映射

切换进程时，内核不仅要换执行线程，还要换这套内存上下文。

这意味着：

- CPU 看到的虚拟地址映射规则变了
- 可能需要切换页表基址

------

## 2）TLB 可能失效或部分失效

TLB（快表）缓存的是“虚拟地址 -> 物理地址”的翻译结果。

如果切换到另一个进程：

- 原进程的地址翻译缓存很多都不再适用
- TLB 命中率会下降
- 之后一段时间内，内存访问会变慢

这也是为什么**进程切换通常比同进程内线程切换更重**。

> 同一进程内两个线程共享地址空间，所以一般不需要切换整套页表，这一点比进程切换轻很多。

------

## 3）内存局部性更容易丢失

进程 A 和进程 B 的代码/数据工作集通常差异更大。
 所以进程切换比线程切换更容易导致：

- 指令 cache 被冲刷热点
- 数据 cache 被污染
- TLB 热项失效

------

# 五、你说的“用户态两个线程或者进程之间切换”到底是什么情况

这里要区分两种东西。

------

## 情况 A：内核线程 / 操作系统线程切换（你平时写 Java 主要是这个）

比如 Java 里的：

- `Thread`
- 线程池线程

这些通常映射到 OS 线程。

只要它们发生调度切换，就一定是：

1. 从用户态进入内核
2. 内核调度器决定切谁
3. 保存/恢复上下文
4. 返回用户态继续执行

所以它**不是真正意义上的“纯用户态切换”**。

------

## 情况 B：用户级线程 / 协程切换（这才可能只在用户态）

比如：

- 运行时自己调度的 green thread
- 协程
- 纤程

这种切换常常只需要：

- 保存少量寄存器
- 切换栈指针
- 更新调度器里的任务状态
- 跳到另一个用户态函数继续执行

它通常**不需要**：

- 进入内核
- 调用 OS 调度器
- 切页表
- 触发内核线程上下文切换

所以它会便宜很多。

但它也不是“零成本”，仍然有：

- 保存/恢复寄存器
- 用户态调度逻辑
- 栈切换
- cache 局部性破坏（虽然通常比进程/内核线程小）

------

# 六、为什么 I/O 密集型场景里“频繁切换”特别伤性能

因为 I/O 密集型程序经常会这样：

1. 用户态发起系统调用
2. 进入内核
3. 发现数据没来，线程阻塞
4. 调度别的线程
5. 数据到了，中断唤醒
6. 再切回来

这里面会反复发生：

- 用户态 ↔ 内核态切换
- 线程阻塞/唤醒
- 调度器介入
- 上下文切换
- cache/TLB 失效

所以高并发 I/O 的优化方向，经常就是减少这些切换：

- 批量系统调用
- 异步 I/O
- 事件驱动（epoll）
- 协程
- 减少锁竞争
- 减少线程数失控

------

# 七、把几种“切换成本”按轻重排个序（通常情况）

一般可以粗略这么理解：

1. **普通函数调用**
    最轻，只是栈帧变化
2. **用户态协程/用户级线程切换**
    轻，主要是用户态保存/恢复少量状态
3. **用户态 ↔ 内核态切换（不阻塞、很短 syscall）**
    比上面重，因为要 trap、切内核栈、做校验、破坏局部性
4. **同一进程内 OS 线程切换**
    更重，因为有调度器 + 线程上下文保存/恢复
5. **不同进程切换**
    通常最重，因为可能还涉及地址空间切换、TLB 影响更明显

这不是绝对数字排序，但理解上基本成立。

------

# 八、一个最容易记住的本质

你可以把“切换成本”记成三类：

## 1）直接成本：必须做的机械动作

- 保存寄存器
- 恢复寄存器
- 切栈
- 调度判断
- 切页表（进程时）

## 2）内存成本：数据搬运/校验

- 用户态和内核态之间拷贝
- 参数检查
- 地址合法性检查

## 3）间接成本：破坏 CPU 局部性

- cache 热点丢失
- TLB 命中下降
- 分支预测失效
- 流水线重新建立

**真正让你感觉“切换很贵”的，往往是第 3 类。**

------

# 九、放到 Java 里怎么理解

你平时写 Java 时，以下场景都可能触发这些成本：

- `synchronized` / `Lock` 竞争激烈 → 阻塞/唤醒 → 线程切换
- 线程池开太多 → 调度频繁 → cache 抖动
- 频繁小 I/O（大量短 syscall）→ 用户态/内核态切换频繁
- 频繁创建线程 → 调度、栈、系统资源管理成本更高

所以 Java 里很多“性能优化”，本质都是在减少切换：

- 少一点线程争抢
- 少一点阻塞
- 用异步/批量化
- 让线程数合理，而不是越多越好
