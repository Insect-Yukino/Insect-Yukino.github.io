实习嘛，最多的就是 CRUD 了，很难接触什么比较高大上的东西【太惨了】。但是在 CRUD 的过程中也遇到了一些查询性能问题，最常见的就是 N + 1 问题。有些数据组装不可避免的需要涉及到其他的一些表。有些接口我使用 apifox 自己本地测试时，响应时间是 5m【汗颜】。太慢了。有一个需求我使用动态拼接 SQL 的方式去处理，产生笛卡尔积，一次请求生成了 300+ 个 SQL【目瞪口呆】。这是我 SQL 处理的太烂了。因此，这里有个常见的操作是事先将需要用到的数据读到内存中，存放在一个 Map 中，这样在下次访问时，直接读取这个 Map 的话时间复杂度直接降到 O(1)，响应时间降到了 500 ms 左右。由此可见，效果确实好。

这里的项目组长不让我使用像 Redis 这样的缓存数据库【无语】，如果有 Redis 这样的缓存数据库的话，将一些数据缓存在 Redis 中确实可以起到很好的效果。当然使用 Redis 不可避免的涉及到网络开销。其中的权衡之道只有具体情况具体分析了。

在使用内存缓存的时候，其实有一点是需要注意的，就是：**如果是全局缓存，当数据修改了，你原来的数据会更新吗？**典型的包括全局 Map，或者是从 `yaml` 配置文件中读取的一些数据。这是需要注意的。

我认为，你是使用 `yaml` 配置的数据，当这些数据读到内存，如果这些数据可修改，应该配合 Nacos 提供的 `@RefreshScope` 在配置文件更新后同步更新内存中的值。如果是需要全局创建 Map 之中缓存结构，我认为应该实现简易的过期机制，或者是使用第三方缓存。
