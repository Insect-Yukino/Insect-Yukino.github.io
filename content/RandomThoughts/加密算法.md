在实习中，有一个数据加密的场景。客户不允许直接将敏感信息在数据库中存储。需要将数据进行加密。

这里没有直接在本地引入一个数据加密算法，而是通过一个第三方加密平台来实现的加密解密功能。俗称 KMS 加密平台。

程序中直接按照接口规范调用 KMS 提供的加密接口，以此来对数据进行加密解密。但是我感觉很有问题，项目中需要处理用户的手机号，保存的时候加密处理，展示的时候需要解密。但是，加密解密操作每次只能对一个数据操作，如果是简单的调用一次接口，延迟尚可以仍受，但是项目中涉及分页等展示大量数据的场景，我感觉频繁的远程调用定然会十分的影响性能，以上是我的推测，奈何我人微言轻，没人听我的，该方案也就这样了。

借着这个机会，我们来了解一下加密解密涉及到的两类算法，然后，我们了解一下数字信封.

### 对称加密算法

#### 核心特点

- **同一把密钥** 用于加密和解密
- **速度快、适合大数据**（文件、数据库字段、通信内容）
- 难点在于：**密钥如何安全分发与管理**

#### 常见算法

- **AES**（国际通用，分组密码，块大小 16 字节）
- **SM4**（国密分组密码，也常见块大小 16 字节）
- **ChaCha20**（流加密，移动端/高性能场景常用）

#### 常见工作模式

对称算法本身（AES/SM4）只能处理固定大小块，需要 mode：

- **ECB**：不推荐（会泄露明文模式）
- **CBC**：传统常用，需要 **IV** + **Padding**
- **GCM/CCM**：现代推荐（**带认证**，同时提供保密性+完整性）

使用对称加密算法，最重要的就是公钥管理，保证公钥不被泄露，就是保证算法的安全性。其中的要点是

1. **密钥怎么安全分发/协商出来（让第三方拿不到）**
2. **密钥怎么安全保存和使用（让泄露面尽可能小）**

##### 1）根本不把对称密钥发给客户端（KMS/网关模式）

公司的 KMS 平台就是使用的这个模式。

- 客户端（或业务服务）只拿到 `keyId` / token
- 真正的对称密钥材料只在 **KMS/HSM** 内部
- 加解密由 KMS 代劳

这样做的好处：

- 密钥不落到业务侧磁盘/配置文件
- 可做权限控制、审计、轮换、吊销
- 泄露面小很多

> 现实里通常是：服务端业务系统拿到 `keyId`，调用 KMS；移动端一般不会直接持有主密钥。

##### 2）最常见的互联网做法：TLS/HTTPS 先把通道密钥协商出来

###### 用 **TLS（HTTPS）** 做密钥协商

- 客户端与服务端通过 TLS 握手
- 过程里会完成：
  - 身份认证（服务端证书）
  - 协商出一把 **会话对称密钥**
- 之后所有 HTTP 数据都用这把会话密钥加密传输

关键点：

- **会话密钥从来不会在网络上以明文形式传输**
- 即使有人抓包，也看不到密钥
- ECDHE 还支持前向安全（服务器私钥泄露也不影响历史会话）

所以：通信层的对称密钥安全，本质是靠 **非对称/密钥交换** 在握手阶段建立起来的。

##### 3）共享密钥通常怎么发？

用非对称方式封装对称密钥

- 客户端有服务端公钥（或证书）
- 客户端随机生成对称密钥（或会话密钥）
- 用公钥加密后发送给服务端（或用 ECDH 协商）
- 双方得到同一把对称密钥，用它加密数据/HMAC

这本质就是对称负责性能，非对称负责分发。

##### 4）共享密钥拿到了，怎么存得安全？

拿到密钥只是开始，真正风险往往在存与用。

###### 服务端常用措施

1. **不把主密钥写死在配置文件**
   - 用 KMS/HSM 托管（最佳）
   - 或用密钥加密密钥（KEK）保护数据密钥（信封加密）
2. **最小权限**
   - 只有特定服务账号能调用密钥
3. **审计与告警**
   - 谁在什么时候用过密钥、用量异常、失败率异常
4. **轮换（rotation）**
   - 定期换 key version
   - 用 keyId + 版本实现平滑轮换
5. **内存安全与日志治理**
   - 不在日志打印明文/密钥/中间串（stringA、明文 key、明文数据）
   - 避免把密钥变成 Java String（不可控驻留），尽量用 byte[] 并及时清零（能做就做）
   - 严控 dump、core、JMX 等运维面

##### 5）防被篡改、被重放、被冒用的补强手段

即使密钥安全，协议层还需要防攻击：

- **HMAC/签名**：防篡改、认证调用方
- **timestamp + nonce / transId 去重**：防重放
- **绑定 deviceId/appId**：做设备级授权与吊销
- **强制 HTTPS**：防窃听、中间人降级/替换等

### 非对称加密算法

#### 核心特点

- 一对密钥：**公钥（public key）** + **私钥（private key）**
- 公钥可以公开，私钥必须保密
- **解决密钥分发难题**：可以把公钥给任何人，不怕泄露

#### 常见算法

- RSA
- ECC/SM2（椭圆曲线，密钥更短，性能更好；国密场景常见 SM2）
- DH/ECDH（密钥协商/交换）

非对称算法，是通过数学计算难题来保证安全性的。通过私钥算出公钥，使用公钥对数据进行加密。但是公钥想要退出私钥却异常困难。

### 签名或摘要算法

上面了解了加密算法，其实我们常见的还有一种签名或摘要算法。加密算法可逆，但是签名或摘要算法不可逆，只能用于验证消息完整性或身份认证。

签名和摘要算法还有有些区别的。

#### 摘要/哈希（Hash）

比如 SHA-256、SM3：

- 输入任意长度数据 → 输出固定长度摘要
- **单向**：不能从摘要还原原文
- 用途：校验完整性、做指纹、做索引、存密码时配合盐等

> 只做 hash 的缺点：任何人都能算，所以它 **不能证明是谁发的**。

#### HMAC

HMAC 不是纯 hash，而是：

> **hash + 密钥** → 消息认证码（MAC）

同样是单向不可逆，但因为引入了 **secret key**，只有持有 key 的双方才能算出同样结果。用于 **完整性 + 身份认证（共享密钥）**

所以 HMAC 的价值不只是完整性，还能回答：**这段数据是不是拥有密钥的人发的？**

### 数字信封

它是一种把大数据用对称加密加密 + 把对称密钥用非对称加密保护的组合方案。它的目的就是：**既安全又高效**。

##### 它解决什么问题？

- 非对称加密（RSA/SM2）**很慢**，不适合加密大文件/大报文
- 对称加密（AES/SM4）**很快**，但密钥怎么安全分发是难点

##### 数字信封里通常有什么？

假设 A 要把数据安全发给 B：

1. **生成一个临时对称密钥**（一次一把，叫 session key / content key）
2. 用这个对称密钥对真实数据加密（AES/SM4）
3. 用 B 的公钥把这个对称密钥加密（RSA/SM2）
4. 把两样东西一起发给 B：

> **数据本身像信件**，对称密钥像打开信件的钥匙，钥匙被放进一个只有收件人能打开的信封。

##### B 怎么解密？

B 收到后：

1. 用自己的 **私钥** 解开 `EncryptedKey` → 得到对称密钥
2. 用对称密钥解开 `EncryptedData` → 得到原文数据
3. （如果带签名）用 A 的公钥验签 → 确认数据没被改、确实来自 A

学习完之前的知识，我们现在讨论一下，如果解决频繁调用 KMS 平台接口的问题。

#### 方案 1：信封加密 + 存脱敏显示字段

##### 写入/更新用户信息时

- 用 KMS `GenerateDataKey` 得到 `plaintextDEK + encryptedDEK`
- 本地用 `plaintextDEK` 对手机号/身份证做 AES-GCM 加密，存密文
- **同时生成并存一份脱敏字符串**（或部分明文/摘要）用于列表展示

**数据库字段建议：**

- `phone_cipher`（密文）
- `phone_mask`（脱敏后的字符串，如 `138****1234`）
- `idno_cipher`
- `idno_mask`（如 `110***********1234`）
- `encrypted_dek`（用于解密的密文数据密钥）
- `dek_key_id / key_version / alg / iv / tag`

列表查询时，直接查 `phone_mask / idno_mask` 展示，**0 次 KMS 调用**，也不需要本地解密。

详情/导出时，需要明文再走取 `encrypted_dek` -> 调 KMS `Decrypt` 得到 `plaintextDEK` -> 本地解密出明文

> 这套方案在企业系统里非常常见，因为列表/分页是最高频路径。

#### 方案 2：只用信封加密，但批量解密时做DEK 复用/缓存

如果不想落库 `mask` 字段，也可以，但列表每次展示仍然要解密明文再脱敏，那就需要优化 KMS 调用。

做法

- **每条记录一个 DEK**：不缓存时，列表 N 条就需要 N 次 `Decrypt(encryptedDEK)`
- 优化：**按 encryptedDEK 去重 + 批量/缓存 unwrap 结果**
  - 同一批结果里，先把 `encryptedDEK` 去重（可能会减少一些）
  - 本地做一个短 TTL 缓存：`encryptedDEK -> plaintextDEK`
  - 列表请求命中缓存时，就不需要每条都调 KMS
