实习时，在公司的项目中看到将 Token 存放到了 MySQL 和 Redis 两个数据库中，这里了解一下为什么这样做。

经过查询可能是因为，**Redis 存在线态 / 快速校验用的 token，MySQL 存长期态 / 可追溯用的 token。**

Redis 和 MySQL 中的 **Token 本身承担了两类完全不同的职责**：

| 职责                           | 适合的存储 |
| ------------------------------ | ---------- |
| 高频鉴权（每个请求都要）       | Redis      |
| 安全审计 / 登录记录 / 强制下线 | MySQL      |

### 二、Redis 里存 Token

1. 极快的鉴权性能
2. 天然支持过期，不需要定时任务
3. 支持立即踢人 / 立即失效

### 三、MySQL 里存 Token

1. 审计 & 合规，Redis 里不可能存储用户登录的历史记录
2. 支持 Refresh Token / 多端登录
3. 系统重启 / Redis 挂了还能恢复

登录时

1. 校验账号密码
2. 生成 token
3. **写 MySQL（持久化）**
4. **写 Redis（带 TTL）**
5. 返回 token

请求鉴权时（高频）

1. 从请求头拿 token
2. **查 Redis**
3. 查到 → 直接放行
4. 查不到 → 401 / 或走降级逻辑

使用这个设计，通常意味着系统：

- 有权限体系
- 有操作审计
- 有踢人 / 封号 / 多端登录
- 有安全合规要求

> Token 的审计和监控是指对 Token 的使用情况进行记录和分析，以便进行安全审计和风险评估。通过将 Token 存储在数据库中，可以方便地记录 Token 的生成、使用、更新和删除等操作日志。这些日志数据可以用于审计 Token 的使用情况，检测异常行为和潜在安全威胁。例如，可以分析 Token 的使用频率、使用时间、使用 IP 地址等信息，发现异常登录和访问行为。此外，数据库还可以提供实时监控和报警功能，及时发现和处理 Token 的异常使用情况，提高系统的安全性和可靠性。
