判断“打印并吞掉”还是“抛出去”，核心看三件事：**这次失败对业务结果是否必须正确**、**上层有没有能力处理/降级**、**你是否在边界层**。

下面用你“调用 ES 查询”的场景，把选择规则讲清楚。

------

## 1）什么时候该抛出异常（让上层决定）

满足任意一条，就优先 **抛出**（或包装成业务异常再抛）：

1. **查询结果是主链路强依赖**
    例如：搜索就是页面核心数据；查不到就不能继续给正确响应。
    → 抛出，让 Controller / 全局异常处理返回错误码或提示。
2. **上层需要根据失败做策略**
    例如：上层要决定“切 MySQL 精准查 / 走缓存 / 降级展示 / 返回默认值”。
    → 抛出（或返回 Result/Optional，让上层显式分支处理）。
3. **需要事务/一致性语义**
    虽然 ES 查询一般不在事务里，但如果失败会影响后续写、扣费、权限判断等关键逻辑
    → 抛出，避免错误结果继续流转。

> 简单说：**失败不应该被悄悄变成“正常空结果”** 时，就抛。

------

## 2）什么时候可以只记录日志然后返回降级结果

满足任意一条，可以 **记录日志 + 降级返回**：

1. **ES 是“增强能力”而非唯一数据源**
    例如：ES 只是加速搜索/推荐；失败可以回退 MySQL（慢点但能用）。
    → catch + fallback。
2. **允许页面“缺一块”**
    例如：热搜榜、猜你喜欢、相似内容，挂了也不影响主功能。
    → 记日志 + 返回空列表/默认数据。
3. **你在服务边界层做兜底**
    例如：对外 API 要稳定：宁可展示“无数据”，也不要 500。
    → 统一降级，但要打点报警。

> 简单说：**业务允许退化** 时，可以吞，但必须“可观测”。

------

## 3）最容易踩坑：把异常吞成“空列表”

对于 ES 查询，很多人这么写：

```
try { ... } catch (Exception e) { log.error(...); return List.of(); }
```

风险：

- 上层以为“真的没有数据”，其实是“ES 挂了”。
- 排查困难，业务指标会被污染（搜索结果突然变少）。

所以：**除非你明确这是降级策略**，否则不要把异常变成空列表。

------

## 4）在你这个 ES 查询场景，推荐的“决策模板”

把查询分成两类：

### A. 业务强依赖（搜索页核心列表 / 管理后台检索必需）

- **抛出异常**（或包装成 `BizException`）
- 由上层统一返回错误码/提示
- 同时记录必要上下文日志

### B. 可降级（推荐、相似、统计、补充信息）

- **catch + 降级**（回退 MySQL/缓存/空结果）
- 但要：
  - **区分日志级别**：可预期的短暂失败用 `warn`，异常栈用 `error`
  - **打点报警**：比如失败率 > 阈值告警

------

## 5）给你一套“建议写法”（既可抛也可降级）

### 方案 1：强依赖——抛出自定义异常

```
try {
    return doEsSearch();
} catch (IOException | ElasticsearchException e) {
    log.error("ES search failed, index={}, query={}, k={}", "content", q, k, e);
    throw new ExternalServiceException("搜索服务暂时不可用，请稍后重试", e);
}
```

### 方案 2：可降级——fallback（同时保留可观测性）

```
try {
    return doEsSearch();
} catch (IOException | ElasticsearchException e) {
    log.warn("ES search degraded, fallback to mysql. q={}, k={}, err={}", q, k, e.toString());
    // metrics.increment("es.search.fail");
    return mysqlFallback(q, k);
}
```
