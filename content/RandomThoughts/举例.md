我感觉实习的好处之一就是可以真正的逼迫自己去动手，比如今天，我在手写 SQL 的时候对 `FROM` 和 `JOIN` 的界限愈发的不清晰，因此我要再总结一下。

我的疑问主要有：

1. `FROM` 和 `JOIN` 的区别
2. `ON` 和 `WHERE` 的区别
3. 为什么 `WHERE` 可以写 `JOIN` 表的字段

## 一、FROM 连接 vs LEFT JOIN 有什么区别？

比如我们看一个 SQL

```sql
SELECT ur.*
FROM user_role ur
LEFT JOIN sys_user u ON u.id = ur.user_id
```

`user_role` 是 **主表**，也被称作 **驱动表**，而 `sys_user` 叫做左连接表。`ON` 是连接条件。

连接方式主要有三种

### 1. INNER JOIN（或逗号连接）

```sql
FROM A, B
WHERE A.id = B.id
```

或者

```sql
FROM A
INNER JOIN B ON A.id = B.id
```

只保留匹配成功的行

------

### 2. LEFT JOIN

```
FROM A
LEFT JOIN B ON A.id = B.id
```

效果：

> 保留 A 的全部数据
>  B 没匹配到时填 NULL

------

# 举例

A 表：

| id   |
| ---- |
| 1    |
| 2    |

B 表：

| id   |
| ---- |
| 1    |

------

### INNER JOIN 结果：

| id   |
| ---- |
| 1    |

------

### LEFT JOIN 结果：

| A.id | B.id |
| ---- | ---- |
| 1    | 1    |
| 2    | NULL |

# 二、ON 和 WHERE 的区别（最重要）

这点非常关键。

------

## 1. ON 是“连接条件”

决定：

> 两张表如何匹配

比如：

```
ON d.stcd = s.stcd
```

表示：

> 用 stcd 做关联

------

## 2. WHERE 是“结果过滤条件”

是在：

> 表连接完成之后
>  再过滤结果

------

# 三、ON 和 WHERE 在 LEFT JOIN 下的本质区别

这是核心。

假设：

```
FROM A
LEFT JOIN B ON A.id = B.id
```

------

### 情况 1 把条件写在 ON

```
LEFT JOIN B ON A.id = B.id AND B.name = 'Tom'
```

效果：

- 只匹配 name = Tom 的 B
- 但 A 仍然全部保留

------

### 情况 2 把条件写在 WHERE

```
LEFT JOIN B ON A.id = B.id
WHERE B.name = 'Tom'
```

效果：

- B.name = NULL 的行会被过滤
- LEFT JOIN 退化成 INNER JOIN

🔥 这是很多人踩坑的地方。

------

# 四、为什么 WHERE 可以写 JOIN 表的字段？

因为 SQL 执行顺序是：

```
1. FROM
2. JOIN
3. ON
4. WHERE
5. GROUP BY
6. HAVING
7. SELECT
8. ORDER BY
```

也就是说：

> WHERE 执行时，JOIN 已经完成了

所以：

```
WHERE s.addvcd LIKE '%xxx%'
```

当然可以。

------

# 五、ON 里能不能用 LIKE？

✅ 完全可以。

例如：

```
LEFT JOIN B ON A.name LIKE '%' || B.keyword || '%'
```

SQL 不限制 ON 只能等值。

只不过：

- 等值连接最常见
- LIKE 会影响索引
- 可能导致全表扫描

# 七、什么时候应该写在 ON？

如果你想：

> 即使 s 不匹配，也保留 d 的数据

应该写成：

```
LEFT JOIN vt_bsc_stinfo_gw s 
ON d.stcd = s.stcd
AND s.addvcd LIKE '%' || #{reqVO.addvcd} || '%'
```

这样：

- d 永远保留
- s 可能为 NULL

------

# 八、一句话总结

| 写在 ON            | 写在 WHERE                     |
| ------------------ | ------------------------------ |
| 控制如何匹配       | 控制最终结果                   |
| 不会破坏 LEFT JOIN | 可能把 LEFT JOIN 变 INNER JOIN |
| 推荐用于关联限制   | 推荐用于整体过滤               |
