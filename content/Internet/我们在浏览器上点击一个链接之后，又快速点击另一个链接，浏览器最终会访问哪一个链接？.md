> 我们在浏览器上点击一个链接之后，又快速点击另一个链接，浏览器最终会访问哪一个链接？

这又是面试官问我的一个问题，当时我其实知道第二次访问的会覆盖第一次访问的，但是我无法使用底层的 TCP 知识解释为什么，真是错过错过。。。

现给出完整的作答。

当发生上述问题时，通常最终访问后点击的那个链接。
因为第二次点击会让浏览器把第一次导航取消，然后第二次点击对应的 URL 作为当前这次页面跳转的目标。

### 1. 浏览器层面发生了什么

当点第一个链接时，浏览器开始一次 **navigation**：

- 解析 URL
- 查缓存
- 发起 DNS / TCP / TLS / HTTP 请求
- 等待响应
- 准备把页面切到新文档

如果这时你又很快点了第二个链接，浏览器通常会认为：

- 用户最新意图是第二个链接
- 第一个导航不再需要

于是浏览器会：

- 把第一个导航标记为 **aborted / canceled**
- 启动第二个导航
- 最终只提交第二个页面到标签页

所以从用户视角看，**页面最后打开的是第二个链接**。

###  2.TCP 为什么看起来会服从后一次点击

**TCP 并不知道你点了哪个链接**。
TCP 只负责可靠传输字节流，真正决定取消哪个请求的是**浏览器和上层协议（HTTP）**，不是 TCP 自己在做二选一。

#### 情况 A：第一个请求还没真正发出去

例如你点第一个链接后，浏览器还在：

- 主线程调度
- 解析 URL
- 查 DNS
- 排队等 socket
- 准备发请求

这时又点第二个链接，浏览器直接把第一次导航任务丢掉。那第一个请求可能压根没有进入网络层，自然只会访问第二个。

#### 情况 B：第一个请求已经发到网络里了

这时浏览器不能把已经发出去的字节收回来，但它仍然可以：

- 不再关心第一个响应
- 关闭对应连接
- 或在 HTTP/2 / HTTP/3 上取消对应 stream
- 然后发第二个请求

于是结果仍然通常是第二个页面生效。

也就是说：

- **第一个请求可能确实到过服务器**
- 但浏览器后来把它取消了
- 即使服务器回了响应，浏览器也可能不再使用它

所以最终访问哪个从页面结果看是第二个，但网络上是否真的发过第一个请求不一定。

TCP 并没有取消 HTTP 请求这种语义，它只有连接和字节流。所以浏览器若想停止第一个请求，常见手段是：

#### 方式 1：直接关闭 TCP 连接

浏览器可以把承载第一个请求的 socket 关掉。

TCP 层面会表现为：

- 发 **FIN**：正常关闭
- 或发 **RST**：强制复位，更像立刻中断

服务器看到连接断了，就知道后面不用继续发了，或者发到一半失败。

#### 方式 2：不关整个 TCP，只取消上层请求

在现代浏览器里更常见，因为常常是：

- 一个 TCP 连接上复用多个请求（HTTP/2）
- 甚至不是单独一个请求一个 TCP 连接

这时不能为了取消一个请求把整个连接都掐掉。于是浏览器会在上层协议里发取消这个流的信号：

- HTTP/2：`RST_STREAM`
- HTTP/3：取消对应 QUIC stream

底下的 TCP（HTTP/2 情况）连接还活着，但其中某个请求逻辑上已经被取消。

所以本质上不是 TCP 决定后一个赢，而是：

- 浏览器决定当前导航目标以最后一次用户操作为准
- 上层协议负责取消旧请求
- TCP 只是把这些控制信息可靠送到对端

为什么浏览器要这么设计？

因为浏览器遵循的是**用户最新操作优先**：

- 你第二次点击，说明你改变主意了
- 标签页同一时刻只能有一个当前导航
- 所以旧导航会被新导航替代

这和很多 UI 系统一样，本质是后来的用户意图覆盖之前的意图。
