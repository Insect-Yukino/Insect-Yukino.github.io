跨域是浏览器独有的安全策略。浏览器为了安全对网页里的 JS 发起的跨站请求加的一套限制。

#### 1）跨域问题怎么产生的：同源策略 + JS 受限

浏览器有 **同源策略（Same-Origin Policy）**，一个网页的脚本只能随意访问同源的资源。

同源 = **协议 + 域名 + 端口**都相同。只要有一个不同，就算跨域，例如：

- `http://localhost:8080` → `http://localhost:8081`（端口不同）
- `http://a.com` → `https://a.com`（协议不同）
- `http://a.com` → `http://api.a.com`（域名不同）

> 这个限制主要针对 **JS 发起的请求读取响应**。

#### 2）为什么浏览器直接访问接口不跨域，但 JS 访问会跨域？

在地址栏直接打开接口（GET）

这是 **导航（navigation）**行为，是由浏览器去加载一个新页面/资源。我们通过浏览器的地址栏发送请求，由浏览器渲染页面。而在已经渲染出的页面获取 JS 资源就不一样了，这时所处的环境是已经被渲染好的页面，是可能有私密信息的，如果这时访问未知 JS 资源，如果这个资源是恶意的，可能就会盗取那些信息，导致信息泄露。

同源策略主要限制的是：**一个页面里的 JS 去读另一个源的响应内容**，而不是手动在地址栏打开某个 URL。

所以：

- 地址栏打开 `http://api.xxx.com/data`：能看到返回 JSON/下载文件 —— **不会触发 CORS 拦截**
- 页面里的 JS `fetch('http://api.xxx.com/data')`：浏览器会检查 CORS —— **不满足就拦截**

> 你可以理解为：浏览器不怕你跳转过去看，但怕网页悄悄读取你的数据。

#### 表单提交 / `<img src>` / `<script src>` 也不一样

这些很多也能跨域发出去（请求能到后端），但：

- JS **不能随意读回响应内容**
- `<img>` 只能当图片用，不能读到像素（除非允许）
- `<script>` 是脚本注入

#### 3）为什么有时候会出现预检请求（OPTIONS）？

如果 JS 请求满足一些条件，就会先发一个 **OPTIONS 预检**：

触发预检的常见情况：

- 使用非简单方法：`PUT/DELETE/PATCH`
- 加了自定义头：`Authorization`, `X-Token`, `Content-Type: application/json`
- 发送 cookies 且 `credentials: 'include'`

流程：

1. 浏览器先发 `OPTIONS` 问后端：我能不能跨域这样请求？
2. 后端若返回允许的 CORS 头，浏览器才会继续真正的请求
3. 不允许就直接在浏览器层面拦截（后端可能已经收到，但前端拿不到响应）

## 实际项目里最常用的三种解决方案

1. **Nginx 反向代理**：
    前端访问 `/api/...` → Nginx 转发到后端 `http://backend:8080/...`（同源）
2. **后端开启 CORS**
3. **开发环境 devServer proxy**

> **发起请求的网页的来源(origin)** 和 **目标接口的来源(origin)** 的 **协议 / 域名 / 端口** 必须完全一致，才叫同源，才不会触发跨域限制。
