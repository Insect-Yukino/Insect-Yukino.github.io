我今天在处理一个导出功能时，后端抛出超时断连异常。。。

我是直接将文件内容写入 response，他报错了，然后向 ChatGPT 求助，他给出的方案是先将数据写到临时文件，再写到 response。我这样做了，成了！

神不神奇？

但是 ChatGPT 给我的答案我不胜满意，看吧，AI 也不是万能的（小声说）。
我在网上查阅之后，其实这个现象可能的原因是：

TCP 底层其实涉及到了两种连接超时：

- 连接超时
- 读写超时

连接超时是 TCP 建立连接的过程发生超时，而读写超时确实发生在建立连接之后。比如进行写操作时，写入不连续很可能会导致写超时。在导出 Excel 文件的场景下，这样会有一部分数据被前端下载（Excel 导出），但是剩下的数据可能会由于后端报异常而缺失。

我上面导出 Excel 的操作，很可能就是由于写入不连续导致触发了超时异常。

而 ChatGPT 给出的方案是先将数据写入操作系统的临时文件，然后通过拷贝的操作直接写入 response，这样由于写入是连续的，上一个响应客户端的数据包相邻着下一个数据包发送，所以不会触发超时异常。

需要注意的是，我们的应用层实现 TCP 连接，是依靠操作系统层的能力的。操作系统实现 TCP 功能是靠 Socket。借着这个机会，我们今天来复习一下 Socket。

## Socket

我们其实可以从两个视角来理解 Socket

- 当我们拿到一个地址 `localhost:8080`，`localhost` 是主机地址，`8080` 是主机暴露的一个端口，一台服务器肯定只有一个网络地址，那如何连接数以千计的网络连接？是怎么精确匹配到我们对应的应用程序？那当然就是靠的是 `port` 端口。我们可以这样理解，`localhost:8080` 就是一个 Socket。在一个服务器上，一个 Socket 就是一个 `port`。
- 而在网络分层中，Socket 就是应用程和网络层的中间层，由 Socket 提供了 TCP 或者 UDP 的真正实现接口。

### Java 封装的 Socket

这里我们简单了解一下 Java 封装的 Socket 实现。

Java 对 Socket 的封装是基于 **TCP/IP 协议栈**，在不同层次上提供了从“原生阻塞 IO”到“高性能多路复用”的多种实现。可以理解为：**Java 在操作系统 Socket 之上做了一层面向对象封装**。

#### 一、最基础的 Socket（BIO 阻塞 IO）

对应类：

- java.net.Socket
- java.net.ServerSocket

Java 的 `Socket` 本质上是：

> JVM → 调用本地方法（JNI） → 调用 OS 的 socket 系统调用

底层依然是：

```java
socket()
bind()
listen()
accept()
read()
write()
```

但 Java 给做了面向对象封装：

```java
ServerSocket server = new ServerSocket(8080);
Socket socket = server.accept();  // 阻塞等待连接

InputStream in = socket.getInputStream();
OutputStream out = socket.getOutputStream();
```

特点

- accept() 阻塞
- read() 阻塞
- 一连接一线程（典型模型）

缺点

高并发下：

- 线程多
- 上下文切换重
- 内存占用大

#### 二、NIO（非阻塞 + 多路复用）

JDK 1.4 引入：

- java.nio.channels.SocketChannel
- java.nio.channels.ServerSocketChannel
- java.nio.channels.Selector

核心思想

> 一个线程管理多个连接（基于 select/epoll）

流程：

```text
Selector 监听多个 Channel
当某个 socket 可读/可写时
通知你处理
```

示意代码：

```java
Selector selector = Selector.open();
ServerSocketChannel server = ServerSocketChannel.open();
server.configureBlocking(false);
server.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select(); // 阻塞直到有事件
    for (SelectionKey key : selector.selectedKeys()) {
        // 处理 accept/read/write
    }
}
```

优点

- 单线程处理成百上千连接
- 基于 OS 的 epoll/kqueue

适合

- 高并发服务器
- RPC 框架
- 网关

#### 三、AIO（异步 IO）

JDK 1.7 引入：

- java.nio.channels.AsynchronousSocketChannel

思想：

> 发起 IO 请求 → 立刻返回 → IO 完成后回调通知

```java
channel.read(buffer, null, new CompletionHandler<Integer, Void>() {
    public void completed(Integer result, Void attachment) {
        // IO 完成回调
    }
});
```

但注意：

- Linux 下仍然基于 epoll 模拟
- Windows 才是真正 IOCP

Java Socket 实现结构大致是：

```text
Java Socket API
        ↓
JDK 内部实现（PlainSocketImpl / NIO Impl）
        ↓
JNI 调用
        ↓
操作系统 socket 系统调用
        ↓
TCP/IP 协议栈
```

也就是说：

> Java 没有重新实现 TCP，它只是封装了 OS 的 TCP。

ChatGPT 讲的整体架构挺详细的。。。

### Socket 连接

我们上面也说过了，Socket 超时其实对应了两种连接超时和读写超时，想要进行详细了解的可以[移步这里](https://zhuanlan.zhihu.com/p/535405145)。接下来我们就对如何设置超时间做一下讨论。

在  Java  `Socket` 中，需要手动指定连接超时

```java
Socket socket = new Socket();
socket.connect(new InetSocketAddress("127.0.0.1", 8080), 3000);
```

第二个参数就是连接超时毫秒数，如果不指定，将取决于系统 TCP 重试。

读超时本质上是设置顶层的 `SO_RCVTIMEO`

```java
socket.setSoTimeout(5000); // 5 秒读超时
```

只对 `read()` 有效，超时会抛出 `SocketTimeoutException`，不影响写操作。

对于写超时，java.net.Socket 没有直接的写超时 API。因为 TCP 写操作通常只是把数据放入缓存区，真正阻塞的是缓冲区满的情况，JDK 没有提供 `SO_SNDTIMEO` 映射。

NIO 设置超时涉及 java.nio.channels.SocketChannel，java.nio.channels.Selector

连接超时可以这样设置

```java
channel.configureBlocking(false);
channel.connect(addr);

while (!channel.finishConnect()) {
    // 自己控制超时逻辑
}
```

配合 

```java
selector.select(timeout);
```

NIO 没有读超时。

> JDK 为什么没有提供写超时控制？问了一下 ChatGPT
>
> C 语言肯定提供了写超时控制，在 C 里可以通过
>
> ```c
> setsockopt()
> ```
>
> 设置：
>
> - SO_SNDTIMEO   （发送超时）
> - SO_RCVTIMEO   （接收超时）
>
> 例如 
>
> ```c
> struct timeval tv;
> tv.tv_sec = 5;
> tv.tv_usec = 0;
> setsockopt(sockfd, SOL_SOCKET, SO_SNDTIMEO, &tv, sizeof(tv));
> ```
>
> 当 send() 因发送缓冲区满而阻塞超过 5 秒返回 `-1`，并设置 `errno=EAGAIN` 或 `EWOULDBLOCK`
>
> 操作系统如何控制写超时？
>
> 当应用调用：
>
> ```c
> send()
> ```
>
> 流程是：
>
> 1. 用户态
> 2. 内核 TCP 发送缓存区
> 3. 网络发送
>
> 如果对端接受慢，TCP 窗口变小，网络拥塞，发送缓冲区满。`send` 会阻塞。
>
> 如果设置了 `SO_SNDTIMEO`，内核内部会：
>
> 1. 启动定时器
> 2. 等待缓冲区可写
> 3. 超时则中断阻塞
>
> 这是 **内核级别的阻塞超时控制**。
>
> 而 JDK 并没有提供写超时。
>
> `Socket` 只封装了 `SO_TIMEOUT`，而没有封装 `SO_SNDTIME`。
>
> 主要原因有：
>
> TCP 写通常不长时间阻塞，因为大部分写只是写入内核缓冲区，很快返回，很少真正卡住。Java 的设计目标又是恶心的跨平台，而 Linux / Windows 等行为略有差异，因此你懂的。Java 更鼓励使用非阻塞 IO 或应用层超时控制

我们这里再说一下写超时，我现在仍然对，写超时是如何触发的？和操作系统发送缓冲区有什么关系？是缓冲区满了才会超时吗？

>  **写超时只会在写操作因为发送缓冲区满而阻塞时才有意义**

如果缓冲区没满，是不会触发写超时的。

TCP 写是应用调用 `write()/send()`，实际上发生：

1. 用户态
2. 内核 TCP 发送缓冲区（SO_SNDBUF）
3. 等待 ACK / 窗口更新
4. 真正发送到网络

也就是说，**`write` 成功不等于对方收到，`write` 成功只是数据被放进内核发送缓冲区**。

当发送缓冲区满了，机会发生 `write` 阻塞。什么情况下会满？

1. 对端接收慢
2. 网络拥堵
3. 对端 TCP 窗口缩小
4. 对端根本不读数据

这时 `write` 需要等待缓冲区腾出空间，也就是等待：

1. 对方 ACK
2. 窗口扩大
3. 数据被发送成功

如果 C 中设置了 `SO_SNDTIMEO = 5 秒` 内核的行为是：

1. `write` 阻塞
2. 启动内核定时器
3. 5 秒内缓冲区没腾出空间
4. `write` 返回 `-1`

即，写超时是等待发送缓冲区可写的超时，不是：

- 等待对方处理超时
- 等待 ACK 超时
- 等待网络完成超时

那么就可以得出一个结论，只要缓冲区有空闲空间，那就不会触发超时。

TCP 窗口又是什么鬼？I don`t know.

**TCP 窗口是 TCP 中用于流量控制和拥塞控制的核心机制**。

TCP 是可靠协议，每发送一段数据，对方都要 ACK 确认，如果丢失要重传，于是 TCP 采用滑动窗口机制，允许在未收到 ACK 之前，可以连续发送多个数据段。窗口大小决定了可以在路上的数据量。

**TCP 实际上有两个窗口控制发送量**：

**接收窗口**：由接收方通告，表示接收方缓冲区还剩多少空间，用于流量控制，在 TCP 报文头里有一个字段 `Window Size`，表示我现在还能接收多少字节。如果接收方缓冲区满了，`Window = 0`，发送方必须停止发送数据。

**拥塞窗口**：由发送方维护，根据网络拥塞情况动态调整，用于拥塞控制，如果网络拥塞，拥塞窗口会变小，发送速率下降。

由此可见，发送方实际可发送数据量：`min(rwnd, cwnd)`。`rwnd` 接收窗口，`cwnd` 发送窗口。

窗口和操作系统缓冲区的关系很大。TCP 的窗口大小约等于接收方 `socket` 接收缓冲区大小。当我们调用 `outputStream#write` 时，数据会：

1. 写入内核发送缓冲区
2. 等待 TCP 发送
3. 等待对方 ACK
4. 对方放入接收缓冲区
5. 对方再通告新的 window

由此可见，这里也可能发生写超时。当对方接收缓冲区满，Window = 0，发送方不能再发，发送缓冲区逐渐被写满。如果一直等不到 window 打开，写操作就会卡住，如果由超时机制，抛写异常超时。

**TCP 窗口的本质**

他是基于 ACK 的动态流量许可机制，他解决两个问题：

1. 防止接收方被淹没（流量控制）
2. 防止网络被打爆（拥塞控制）

## HTTP 客户端

其实在真实项目中，是 HTTP 客户端中的超时，接下来让 ChatGPT 总结一下 HTTP 客户端如何配置。

HTTP 客户端的超时一般分为 **3 类**（不同库名字略有差异）：

1. **连接超时（connect timeout）**：建立 TCP 连接最多等多久
2. **读取超时（read/socket timeout）**：连接已建立后，等服务器响应数据最多多久
3. **连接池获取超时（connection request timeout）**：从连接池里取连接最多等多久

> 在微服务里，这三个都必须设置，否则很容易线程堆积、请求雪崩。

下面按常见 Java 客户端说明。

### 一、JDK 原生 HTTP

#### 旧版 `HttpURLConnection`

类：

- java.net.HttpURLConnection

```java
URL url = new URL("http://example.com");
HttpURLConnection conn = (HttpURLConnection) url.openConnection();

conn.setConnectTimeout(3000); // 连接超时
conn.setReadTimeout(5000);    // 读取超时
```

#### JDK 11+ `HttpClient`

类：

- java.net.http.HttpClient

```java
HttpClient client = HttpClient.newBuilder()
        .connectTimeout(Duration.ofSeconds(3))
        .build();

HttpRequest request = HttpRequest.newBuilder()
        .uri(URI.create("http://example.com"))
        .timeout(Duration.ofSeconds(5))  // 整体请求超时
        .build();
```

说明：

- `connectTimeout`：连接超时
- `request.timeout()`：**整个请求的最大耗时（包含连接+读取）**

### 二、Spring 项目最常见：RestTemplate

类：

- org.springframework.web.client.RestTemplate

默认是**没有超时的**，必须手动设置。

#### 基于 SimpleClientHttpRequestFactory

```java
SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
factory.setConnectTimeout(3000);
factory.setReadTimeout(5000);

RestTemplate restTemplate = new RestTemplate(factory);
```

#### 基于 Apache HttpClient

底层：

- Apache HttpClient

```java
RequestConfig config = RequestConfig.custom()
        .setConnectTimeout(3000)
        .setSocketTimeout(5000)
        .setConnectionRequestTimeout(2000)
        .build();

CloseableHttpClient httpClient =
        HttpClients.custom()
                .setDefaultRequestConfig(config)
                .build();

RestTemplate restTemplate =
        new RestTemplate(new HttpComponentsClientHttpRequestFactory(httpClient));
```

三个超时分别是：

| 参数                     | 含义               |
| ------------------------ | ------------------ |
| connectTimeout           | 建立 TCP 连接      |
| socketTimeout            | 等待服务器响应数据 |
| connectionRequestTimeout | 从连接池取连接     |

### 三、现代项目推荐：WebClient（响应式）

类：

- org.springframework.web.reactive.function.client.WebClient

底层通常是：

- Reactor Netty

```java
HttpClient httpClient = HttpClient.create()
        .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 3000)
        .responseTimeout(Duration.ofSeconds(5));

WebClient webClient = WebClient.builder()
        .clientConnector(new ReactorClientHttpConnector(httpClient))
        .build();
```

### 四、OkHttp

库：

- OkHttp

```java
OkHttpClient client = new OkHttpClient.Builder()
        .connectTimeout(3, TimeUnit.SECONDS)
        .readTimeout(5, TimeUnit.SECONDS)
        .writeTimeout(5, TimeUnit.SECONDS)
        .build();
```

支持写超时

### 五、生产环境建议配置

内部微服务：

- connectTimeout：1~3 秒
- readTimeout：3~5 秒
- connectionRequestTimeout：1~2 秒

调用外部第三方：

- connectTimeout：3 秒
- readTimeout：5~10 秒

### 六、为什么一定要设置超时？

如果不设置：

- 下游卡死 → 线程一直阻塞
- 线程池被耗尽
- 整个服务不可用
- 触发雪崩效应
