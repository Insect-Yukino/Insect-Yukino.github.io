之前见到一个奇怪的现象，使用数据库主键使用自增序列，当插入数据失败，事务回滚，下次再插入时。自增序列是从归滚事务那里接着开始的。抱着求知的心态，我打开的 ChatGPT。

因为**序列的设计目标**是高并发下快速、唯一地发号。

### 1. 序列本质上是发号器，不是表数据

事务主要保护的是：

- 表里的行
- 行上的字段修改
- 一组 SQL 的原子性

而 `nextval()` 做的事更像：

> 给我发一个从未发过的新号码

这个动作一旦发生，数据库通常就认为这个号码已经被占用了。

### 2. 如果序列跟着事务回滚，会有并发冲突

这是最核心的原因。

假设有两个事务：

- 事务 A 取到序列值 `100`
- 事务 B 取到序列值 `101`

然后：

- A 回滚
- B 提交

如果允许 A 回滚后把 `100` 还回去，那接下来另一个事务 C 又可能拿到 `100`

这会带来几个问题：

#### 问题一：实现复杂

数据库要精确跟踪：

- 哪个号被哪个事务拿了
- 哪些号还能回收
- 哪些号不能回收

并发一高，管理成本非常大。

#### 问题二：容易阻塞

为了保证可回滚且不重复，数据库就需要对序列分配做更强的锁控制。

这样会导致：

- 多事务争抢
- 发号性能下降
- 吞吐量变差

#### 问题三：可能破坏唯一性和顺序

如果一个已经发出去但后来回滚的号被重复利用，在某些场景下会让系统更难推断谁先谁后，也增加重复使用旧号的风险。

所以数据库通常直接选一种更简单稳妥的策略：

> **号一旦发出，就不回收。**

### 3. 数据库更看重唯一和高性能，不看重连续

序列一般只保证：

- 唯一
- 大体递增

**不保证连续**。

这就是数据库的取舍：

- 连续，不重要
- 唯一，重要
- 并发性能，重要

所以即使事务失败，导致某些号码最终没被表数据使用，数据库也接受这种空洞。

### 4. 回滚的是业务数据，不是资源消耗痕迹

你可以把 `nextval()` 理解成一种资源消耗。

就像：

- 打印机吐出一张编号纸
- 银行叫号机吐出一个号码
- 仓库分配一个流水号

即使后面业务没办成，这个号码也已经发出去了，不会塞回机器里给下一个人再用。

事务回滚回的是：

- 这条记录不算数
- 这次更新撤销掉

而不是：

- 时光倒流，把已经发出去的号再收回

------

## 5. 这能显著降低数据库内部复杂度

如果序列也参与事务回滚，数据库内部要处理很多麻烦问题：

- 回滚时是否归还号码
- 归还后是否会被其他事务拿到
- 已经缓存给某个会话的一批号码怎么办
- 崩溃恢复时如何恢复“哪些号可重用”

这些都会让序列机制变复杂、变慢、变脆弱。

而当前做法很简单：

- `nextval()` 原子地取一个新值
- 取完就结束
- 不管事务后续成功还是失败，这个值都不再重复使用

实现简单，性能也好。

------

## 6. 所以才会出现“跳号”

比如：

1. 事务 A 取到 `100`
2. 事务 B 取到 `101`
3. A 回滚
4. B 提交
5. 下次取到 `102`

最后表里可能只有：

- 101
- 102

而没有 100

这不是异常，而是序列机制的正常结果。

------

## 7. 一句话抓住本质

**因为序列的首要目标是“唯一、快速、低冲突地发号”，而不是“像表数据一样参与事务回滚”。**

所以数据库宁愿接受：

- 号码跳号
- 不连续

也不愿意为了“回滚后归还号码”付出巨大的并发和实现成本。

有区别，而且 **MySQL 的 `AUTO_INCREMENT` 也会出现和序列一样的“跳号、回滚不回退”现象**。
 只是它们的实现方式和使用方式不完全一样。

------

## 先说结论

### 相同点

- 都是为了给新行生成唯一编号
- 都**不保证连续**
- 事务回滚后，已经分配出去的值通常**不会回退、不会复用**
- 所以都会出现 **跳号**

MySQL 官方文档明确写了：
 **一旦为 `AUTO_INCREMENT` 列生成了值，这个值就不能回滚；即使语句没完成，或者事务回滚了，这些值也会丢失且不再重用。** 

------

## 1. 自增序列 和 MySQL 自增键的核心区别

### 序列（sequence）

像 PostgreSQL / Kingbase 常见的：

```
select nextval('system_role_menu_seq');
```

它是一个**独立对象**，专门负责发号。
 你可以：

- 单独取号
- 手动设置号段
- 多张表共用一个序列
- 在插入前先拿到 ID

也就是说，**序列和表是分离的**。PostgreSQL 官方把 sequence 描述为专门生成连续序号的独立对象，`nextval()` 是多用户安全的取号方式。

------

### MySQL `AUTO_INCREMENT`

它不是一个独立 sequence 对象，而是**表中某一列的属性**：

```
id bigint primary key auto_increment
```

它跟表绑定在一起，通常在 `INSERT` 时由 MySQL / InnoDB 自动分配。官方文档说明，向 `AUTO_INCREMENT` 列插入 `NULL` 或 `0` 时，会自动设置为下一个值。

也就是说，**MySQL 自增键更像“表内置发号机制”**。

------

## 2. 它们的实现差异

### 序列

- 独立存储
- 通过 `nextval()` 明确取号
- 取号和插入可以分开
- 非常灵活

### MySQL 自增键

- 跟表绑定
- 多数情况下在 `INSERT` 时自动分配
- 你一般不会先单独“取一个号”再插入
- 对应用层来说更省事

------

## 3. MySQL 自增键是不是靠锁实现的

**部分是，但不能简单理解成“因为加锁，所以就会回滚”。**

InnoDB 官方文档说，`AUTO_INCREMENT` 分配会涉及一种特殊的 **AUTO-INC lock**。在简单情况下，一个事务往带 `AUTO_INCREMENT` 列的表插入时，其他事务可能需要等待，以保证分配行为正确。

但这个锁的作用主要是：

- 协调并发插入
- 保证分配出的自增值不冲突
- 在某些模式下尽量保持语句级连续性

它**不是**为了让自增值参与事务回滚。

------

## 4. MySQL 自增键会不会像序列一样“回滚不回退”

**会。**

这是和 sequence 很像的地方。

官方原文含义就是：

- 事务里生成了 auto-increment 值
- 后面即使回滚
- 这些值也算“消耗掉了”
- 不会再重新使用
- 表里就可能出现空洞、跳号 

例如：

```
begin;
insert into t(name) values ('a'); -- 分到 id = 100
rollback;

insert into t(name) values ('b'); -- 可能直接是 101
```

最后表里只有：

- 101

没有 100。
 这和你前面说的 sequence 现象本质上一样。

------

## 5. 为什么两者都会这样

因为两者都优先保证：

- 唯一性
- 并发性能
- 实现简单可靠

而不是保证“编号绝对连续”。

PostgreSQL 文档也明确说明，`nextval` 不回滚，是为了避免并发事务取号时互相阻塞。

MySQL `AUTO_INCREMENT` 也是类似取舍：
 宁可允许跳号，也不把“发号”设计成跟业务数据一样可回滚，否则并发和实现复杂度都会明显上升。

------

## 6. 一个很实用的对比

### PostgreSQL / Kingbase sequence

你可以先这样：

```
select nextval('seq');   -- 先拿号
insert into t(id, name) values (?, ?);
```

所以你会很明显地看到：

- “号拿到了”
- “但表里不一定有数据”

------

### MySQL auto_increment

通常你直接这样：

```
insert into t(name) values ('a');
```

ID 是插入时自动分配的。
 所以你不太会单独看到“取号”这个动作，但底层仍然会发生“分配了一个值但最后没落库”的情况。

------

## 7. 哪个更适合什么场景

### 用 sequence 更适合

- 你想插入前先拿到 ID
- 多表共用一套编号
- 想更精细控制编号生成

### 用 MySQL auto_increment 更适合

- 单表主键生成
- 应用层想简单一些
- 不想自己显式处理 sequence

------

## 8. 最后给你一个最容易记住的结论

**区别：**

- `sequence` 是独立发号器
- MySQL `AUTO_INCREMENT` 是表字段自带的发号机制

**共同点：**

- 都会跳号
- 都不保证连续
- 都通常不会因为事务回滚而把已经分配的编号还回去 

所以你问的这个核心问题，答案是：

> **会，MySQL 的自增键也会出现和自增序列一样的现象：事务回滚后，自增值通常不会回退，表中可能出现空洞。**
