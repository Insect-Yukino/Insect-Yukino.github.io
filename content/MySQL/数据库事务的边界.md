声明情况下事务才能生效？

在绝大多数关系型数据库里，**事务的边界核心是：同一个数据库连接**。

##### 同一个连接：一定是同一个事务

- 只要在同一个 JDBC `Connection` 上执行多条 SQL，它们天然可以处在同一个事务里。

##### 同一个库不同 schema：仍然是同一个事务

- schema 只是命名空间：`schemaA.t1`、`schemaB.t2`
- 只要都在 **同一个 database**、**同一个连接** 上执行，事务覆盖它们完全没问题。

> 但是，不要闲的没事，把数据库的每一个 schema 都分成一个 JDBC 连接，这样他们不是一个连接，自然不能保证事务的边界

##### 不同 database：默认不可能是同一个数据库事务

- PostgreSQL/Kingbase 一般是一连接对应一个 database，不能在一个事务里同时操作 `db1` 和 `db2`。
- 要跨库原子一致，只能靠 **分布式事务（2PC/XA/JTA）** 或业务补偿。

##### 不同数据库实例/不同机器：更不可能

- 只能分布式事务或最终一致。

### Spring 事务什么时候才会生效？

Spring 的 `@Transactional` 不是数据库特性，它是 **在应用层用 AOP 把开启/提交/回滚事务这套流程包起来。生效必须同时满足这些条件：**

###### 条件 A：必须经过 Spring 代理（AOP）

也就是调用必须走到代理对象上。

常见失效场景：

- **同类内部自调用**：`this.method()` 调用不会走代理 ⇒ `@Transactional` 不生效
- 方法不是 public（默认基于代理时常见限制，尤其 JDK 动态代理）
- Bean 没被 Spring 管理（new 出来的对象）

###### 条件 B：必须有正确的事务管理器

- 单数据源：`DataSourceTransactionManager`
- 多数据源：每个 DataSource 对应一个事务管理器；不指定的话可能用错

如果在一个 `@Transactional` 方法里同时用了两个 DataSource：

- 默认只会对 **一个** 数据源开事务
- 另一个数据源的 SQL 可能是自动提交状态 ⇒ 看起来就像事务失效

###### 条件 C：必须在同一个线程、同一个连接上工作

Spring 通过 `ThreadLocal` 绑定事务上下文。

所以：

- 在事务里开新线程 / `@Async` / 线程池执行 ⇒ 新线程拿不到原事务上下文 ⇒ 不在同一事务
- 用了多个连接（比如切换数据源、或者手动 `DriverManager.getConnection`）⇒ 也不在同一事务

###### 条件 D：异常类型要触发回滚（默认规则）

Spring 默认：

- `RuntimeException` / `Error` 才回滚
- `Checked Exception`（受检异常）默认 **不回滚**，除非你配置 `rollbackFor`
