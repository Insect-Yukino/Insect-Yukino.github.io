我实习的时候做一个功能，需要建一张表，这张表我是用了自增主键。然后又又又被 leader 谈了，他说不能用自增主键，使用自增主键会让数据迁移变的困难。然后我就了解了以下。

在我以往的学习中，自增主键常常被描述成高效的代名词。但是在实际生产环境里却被告之不用自增主键。

自增主键的好处我认为主要是对于插入比较友好，因为是顺序插入，效率肯定是比较高的。但是自增主键确实在很多 **真实场景**下会带来 **架构约束和运维风险**。常见原因有这些：

### 1) 分布式/多实例写入时不好用

一旦有：

- 多个服务实例同时写
- 读写分离 + 可能的主从切换
- 分库分表
- 甚至多主/双活

**自增 ID 需要一个单点递增源**。拆库后每个库都从 1 开始就会冲突；做全局自增又会引入集中式 ID 服务或强一致协调，复杂度上升。

### 2) 数据迁移、合库、离线导入容易冲突

生产经常发生：

- 多系统数据合并
- 历史库迁移
- 离线导入/回灌
- 灾备库切换后回放

自增主键在来源多的情况下 **很容易撞号**，而且修复成本高。

### 3) 安全与业务风险：ID 可预测

自增是连续的，意味着：

- 通过 `/detail?id=10001` 很容易被人顺手枚举到别人的数据
- 暴露业务量、增长速度（信息泄露）

### 4) 写入热点与性能问题

连续递增在一些实现下会导致高并发时出现锁竞争/争用（尤其是某些存储引擎或配置下）MySQL InnoDB 里自增常常性能还不错，但在极高并发 + 特定表设计/事务模式下仍可能成为瓶颈点之一。

> MySQL 数据库在操作自增主键时会上锁，上锁不可避免的会引发锁竞争，导致写入性能变差。

### 5) 业务上更需要可追踪/跨系统唯一的 ID

很多生产系统希望主键能：

- 全局唯一（跨库、跨环境、跨系统）
- 在日志、MQ、链路追踪里直接用
- 支持离线生成（客户端/边缘节点也能先生成）

这类需求更适合：**Snowflake（雪花ID）/UUID/ULID/数据库序列+号段** 等方案。

> 我实习的公司，使用的是 MyBatis-Plus，他们的主键策略是使用框架自带的：
>
> ```java
> @TableId(type = IdType.ASSIGN_ID)
> ```
>
> 这个方案是雪花算法，他算法生成的 ID 是有序的，插入查询的效率不错，但还是有点问题。就是 **时间戳回拨**。
>
> 当然还有 `IdType.ASSIGN_UUID`，这是 UUID 策略，当然，UUID7 之前的版本，生成的 ID 是无序的，性能不怎么好，但是 UUID7 在网上说的是可以胜任，他是有序的。
>
> 收到这两个 ID 生成策略，那他们的回填时机也应该了解一下。
>
> `IdType.ASSIGN_ID` / `IdType.ASSIGN_UUID` 肯定是插入前回填，由 MP 执行，他不是数据库自增 ID，数据库自增是执行 Insert 插入之后才会给本条数据的主键赋值，我们想要在插入后拿到这个 ID 可以通过 JDBC 的 `getGeneratedKeys` 或 MyBatis 的 key 回填机制 把 id 写回实体。
>
> 但 MP 的 `IdType.ASSIGN_ID` / `IdType.ASSIGN_UUID` 是应用层的，他由 MP 的某个组件执行 `set` 操作，在 Insert 之前我们就可以通过原来的对象 `get` 到 ID。

那生产里就完全不用自增吗？

我不知道，但是我问 ChatGPT 他说：

在一些 **单库单表、内部系统、数据不出域** 的场景，自增主键依然很常见、也很好用（简单、占用空间小、写入顺序友好）。

更典型的生产折中是：

- **表内仍用自增/sequence 做内部主键**（便于索引、join）
- **对外/跨系统用业务 ID（雪花ID/UUID/ULID）**（避免可预测、保证全局唯一）

## 逻辑删除

在我实习的公司，逻辑删除非常常见，但是一开始我在处理的时候非常难受，因为涉及一个场景，先删除一条数据，然后之后有可能这条数据又新增回来了，主键 ID 使用的是业务 ID。

逻辑删除是通过 `deleted` 字段的值来判断该条数据在逻辑上是否被删除，通常 `0` 代表没有被删除，`1` 代表数据被删除了。他只是在业务层面消失了，因为查询时总会加上 `WHERE deleted = 0`，但是在数据库层没有，因此你没在判断该主键是否存在时就执行 Insert 操作很可能会报主键冲突错误。。。

而和业务非关联的主键很完美的解决了这个问题，比如使用 `IdType.ASSIGN_ID`，因为每次主键 ID 都是不一样的，因此是不会出现和  `deleted = 1` 的之前的数据的冲突。但是这是在字段上没有唯一约束的情况下，如果字段上有唯一约束，那非业务字段可能就没什么用了。

上面是我的一些感悟，接下来让 ChatGPT 选手来给我们做一个比较系统的讲解。 

### 1) 业务主键（有业务含义）为什么不总是够用？

业务字段做主键的优点是直观、可读（例如：设备编号 `hte_code`、合同编号 `contract_no`）。但在生产里经常遇到这些痛点：

- **业务会变**
   规则一改：编号长度、编码规则、前缀、部门编码组成方式变化……一旦业务主键变了，所有外键、关联表、缓存 key、日志引用、接口参数都会跟着改，代价巨大。
- **业务字段不一定稳定唯一**
   可能出现：历史数据冲突、人工录入重复、不同来源系统编号不统一、临时号/占位号。
- **业务字段往往更长**
   用它做主键意味着所有二级索引都要存一份主键值（像 InnoDB），长主键会让索引膨胀、join 成本更高。
- **业务含义的字段可能涉及隐私/安全**
   直接暴露身份证号/手机号/设备真实编号等做主键，会让内部数据结构外泄。

### 2) 那为什么还要加一个没业务语义的主键？

这个无业务语义的主键（也叫 **代理键/Surrogate Key**，如 `id`）主要是为了工程稳定性和性能：

- **稳定：业务改规则不影响关联关系**
   业务字段改动时，只要更新业务字段本身，表间关联仍靠 `id` 不动。
- **性能：短主键更适合做 join 和索引**
   数字型/短字符串主键让索引更小，join 更快，外键表存储更省。
- **解耦：方便多系统汇聚与数据迁移**
   合库/导入时，业务编号可能冲突，但 `id` 可以用全局唯一策略生成，先把数据落下来再做业务映射。
- **便于做版本化/历史**
   同一个业务编号可能会出现多条记录（历史版本、状态流转记录、软删除后重建等）。这时可以让 `id` 唯一，而业务编号只要求在某个条件下唯一。

### 3) 最常见的设计

- `id`：作为主键（技术主键，稳定、短、join 用）
- `business_key`：加 **唯一约束**（保证业务唯一性）

举例（很常见）：

- 资产表：`id` 主键 + `asset_number` 唯一
- 设备表：`id` 主键 + `hte_code` 唯一
- 用户表：`id` 主键 + `username`/`mobile` 唯一（可能还加条件唯一）

### 4) 什么时候更适合业务字段直接做主键？

通常是业务键满足：**短、稳定、永不变、更改概率极低、全局唯一且不敏感**。比如某些标准化编码、国家/国际标准号，且系统规模不大、关联简单。

自增 `id` **并不一定** 降低性能——在很多单库单表场景里，它对 **B+Tree 主键（聚簇索引）插入** 反而是友好的（基本是追加写）。但在一些生产负载下（高并发、批量插入、频繁删最近数据等），它确实可能让性能变差。

#### 1）从插入（INSERT）角度：为什么可能变慢

##### A. 自增值生成会出现争用/串行化

很多数据库实现自增都需要一个全局递增点（锁/闩/序列对象）来保证不重复。

- 并发非常高时，大家都在抢这个点，会出现 **自增锁/序列争用**，导致插入吞吐下降。
- MySQL InnoDB 里常见的是 **AUTO-INC lock / mutex** 的竞争（尤其批量 insert、insert…select、并发多时更明显）。

> 现象：CPU 不高但 TPS 上不去、等待在自增相关锁上、插入延迟抖动。

##### B. 主键索引最右端热点（Rightmost hot page）

自增主键意味着新行永远插到 B+Tree 的最右侧叶子页附近：

- 优点：页分裂少、写入顺序好（对单机/低并发很爽）
- 缺点：并发高时，大家都在写同一小片区域，会产生 **页级闩锁竞争**（热点页）

> 主键索引最右端热点（也叫 **Rightmost Hotspot / 右侧热点 / 递增主键写热点**）指的是：**主键按递增生成（自增、序列、按时间递增的雪花等）时，所有新插入的记录都会落在 B+Tree 主键索引的最右叶子页**，导致并发写入时大家都在争抢同一小片数据结构，从而出现锁/闩（latch）竞争、页分裂、吞吐下降。
>
> ##### 1）它本质上在热什么
>
> 以 InnoDB/PG/Kingbase 这类 B+Tree 为例：
>
> - 主键索引是按 key 有序的 B+Tree
> - 新 key 永远最大 ⇒ 新行永远插到 **最右叶子页**
> - 高并发 insert ⇒ **很多线程同时改同一页**（以及其父节点路径）
> - 这会带来：
>   - **页级闩/锁竞争**（保护 B+Tree 页的并发修改）
>   - **页分裂**（叶子页满了要 split）
>   - **日志/刷盘压力集中**（redo/WAL、buffer 写回更集中）
>
> 结果就是：CPU 明明有很多核，但大家排队抢右端页，QPS/吞吐上不去，延迟抖动变大。
>
> ##### 2）典型表现
>
> - 插入吞吐在并发升高后不线性提升，甚至下降
> - p99 延迟抖动明显
> - 数据库里出现明显的索引相关等待（不同库名称不同，比如 latch 等待）
> - 监控里 buffer pool 命中率还行，但 insert 仍慢（因为瓶颈是争抢同一页，不是 IO）
>
> ##### 3）为什么递增主键更容易触发
>
> - **随机主键（UUID/纯随机）**：会把插入打散到各个叶子页，争用分散，但代价是：更频繁的页分裂、更差的局部性、索引更碎、范围查询更差
> - **递增主键（自增/序列/时间递增雪花）**：局部性好、范围查询友好、索引更紧凑，但并发写会集中在右端 ⇒ 热点
>
> 所以它是一个典型的 **读写局部性 vs 写入并发热点** 的权衡。
>
> ##### 4）怎么缓解
>
> ###### A. 降低同一页的并发冲突
>
> - **批量写入**（batch insert），减少 B+Tree 修改次数
> - 让写入更均匀：**分库分表 / 按业务维度拆表**（热点从一棵树拆成多棵树）
>
> ###### B. 打散 key
>
> - **雪花 ID**：虽说整体递增，但低位包含序列，仍然基本单调递增，右端热点依然可能存在（只是比纯自增稍好/或相近）
> - **带散列的前缀/分段号**：例如 `shard_id + 自增`，让不同 shard 落在不同 key 区间（常用）
> - **UUID v7 / ULID**：时间有序但有随机性（不同实现效果不同），一般能比纯自增更分散一点，但仍可能偏右
>
> ##### C. 物理层/引擎参数层面的优化（因数据库而异）
>
> - 提高缓冲与并发能力、减少分裂成本、减少争用（不同数据库有不同参数/机制）
> - 但 **根治通常还是从 key 分布/写入模式入手**

##### C. 批量插入时更容易碰到锁模式差异

一些场景下（例如某些 SQL 语句形式、隔离级别、复制方式等），会导致自增分配更保守，进一步降低并发度。

#### 2）从删除（DELETE）角度：自增本身不必然慢，但会放大某些热点

删除的开销主要来自：

- 删除要修改索引结构（主键索引 + 所有二级索引）
- MVCC 下还涉及版本链、purge/vacuum（不同数据库叫法不同）
- 可能产生碎片、页空洞，影响后续扫描/缓存命中

自增主键会在下面两类删除模式下更容易显得慢：

##### A. 按时间删最近数据会造成同一批页反复被改

很多业务删数据是按时间窗口：

- 删最近产生的临时数据
- 删最新批次导入的错误数据
- TTL 过期清理按创建时间/ID 范围删

而自增 `id` 与时间高度相关（越大越新），于是：

- 你 `DELETE WHERE id > 某值` 或按创建时间删，往往集中命中 **B+Tree 右侧那一段页**
- 这会让删除也变成热点页修改，和插入抢同一片区域 → 闩锁/IO 更集中

##### B. 频繁删除会导致索引空洞/碎片，影响后续查询

即使自增插入本来很顺滑：

- 如果你又频繁把某些区间的行删掉，会留下大量空洞
- 范围查询/扫描可能读更多页，缓存命中变差
- 有些库需要额外的维护（purge/vacuum/optimize）才能回收空间、恢复性能

> 注意：碎片不是自增独有，但自增 + 按新数据删更容易集中发生在同一片叶子页，让性能抖动更明显。
