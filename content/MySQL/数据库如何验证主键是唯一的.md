数据库保证主键唯一，靠的不是每次插入先去查一遍，而是靠 **主键约束 + 唯一索引（通常是 B+Tree）** 在写入时做强制校验。

#### 1）主键唯一靠什么结构保证？

当创建表：

```sql
PRIMARY KEY (id)
```

数据库会做两件事：

1. **建立主键约束（constraint）**
2. **创建一个唯一索引（unique index）**

这个唯一索引的索引键就是 `id`。因此主键唯一本质上就是唯一索引唯一。

#### 2）它是怎么判断是否重复的？

插入一条新记录 `id = X` 时，数据库会在 **唯一索引** 里查找键值 `X`：

- 如果索引里已经存在键 `X` → 直接报错
- 如果不存在 → 才允许把索引项插进去，再插入数据行

由于索引是有序结构（常见 B+Tree）：

- 查找 `X` 是 `O(logN)`（很快）
- 判断重复就是查到没查到的问题

> 不是全表扫描；是走索引定位。

#### 3）并发场景怎么保证不会同时插入同一个 id？

关键是 **锁 + 事务 + 索引层面的并发控制**。

两个事务 T1/T2 同时插 `id = X`：

- T1 先在唯一索引位置插入/锁定 `X` 对应的索引范围（或索引记录）
- T2 也来插 `X`，会在索引层面发现冲突：
  - 要么被阻塞等锁
  - 要么直接报唯一冲突（取决于数据库实现/隔离级别/语句）

最终效果：**最多只有一个成功**。

不同数据库用的机制可能是：

- **索引记录锁 / gap lock / next-key lock**（MySQL InnoDB 常见）
- **MVCC + unique index 原子检查**（PostgreSQL/Kingbase 类似）
- **latch/lock + unique check**（Oracle/SQL Server 也有类似思路）

但共同点都是：**唯一索引层面做原子性的检查+插入**，并发下不会漏。
