## 网络 IO

### 同步阻塞 IO

线程调用 `read` 后，如果数据还没准备好，线程会一直阻塞，直到内核把数据准备好并拷贝到用户态，`read` 才返回。

线程在整个 IO 过程中被挂起，当服务器并发量大时，需要很多线程，造成极高开销。毕竟服务器总不能直接拒绝处理正常的 TCP 链接吧

### 同步非阻塞 IO

线程调用 `read` 后，如果数据没有准备好，内核立即返回一个错误，如 `EAGAIN`，线程不会阻塞。于是用户线程需要反复调用 `read` 去检查数据是否到了。

这样线程不会卡在一次 `read` 中，但是需要不断轮询，会导致 CPU 空转，效率较低，比较鸡肋。

### IO 多路复用

它是指，用户线程先通过 `select`、`poll`、`epoll` 等机制，把多个 socket 注册到内核中，然后**阻塞这些多路复用系统调用上**。

当每个或某些 socket 就绪后，这个系统调用才返回，用户线程再对就绪的 socket 调用 `read`/`write` 进行真正的数据读写。

IO 多路复用的关键点是，**一个线程可以同时监听多个 IO 连接**，线程不是不停的自己轮询每个 `read`，而是由内核帮我们统一等待哪些 fd 就绪。当 fd 就绪后，线程再去处理。

值得注意的是，IO 多路复用通常属于**同步 IO**，因为用户线程调用 `select/epoll_wait` 时，仍然会阻塞等待，就绪后，线程还要自己再调用 `read` 把数据从内核拷贝到用户空间。也就是说，**数据拷贝这一步仍然是同步完成的**。

真正的**异步 IO** 是指，用户线程发起 IO 请求后立即返回，内核自己完成等待数据+拷贝数据，全部完成后再通知用户线程

对于**单个连接**而言，IO 多路复用不一定比单线程阻塞 IO 更快。但是对**高并发网络服务器**而言，它可以显著提高了整体吞吐和资源利用率。因为它减少了线程数量、上下文切换和线程创建销毁成本。

## 一个网络请求的一生

上文提到了内核会准备数据，而网络 IO 阻塞的就是这部分时间。这一时间让我很困惑，为什么用户线程都收到 HTTP 请求了，还不知道数据有没有准备好？这又进一步引发了我的另一个思考，我们平常十分容易开发的接口，是怎么收发 HTTP 的？

### 客户端准备发起请求

假如浏览器访问

```http
http://example.com/api/user
```

客户端首先会做一下事情

#### 1. 解析 URL

比如：

- 协议：`http`
- 主机：`example.com`
- 端口：默认 80
- 路径：`/api/user`

#### 2. DNS 解析

客户端要先知道 `example.com` 对应哪个 IP。所以会去做 DNS 查询，得到类似给机器看的网络 IP `93.184.216.34`

### 客户端发起 TCP 连接

从现在开始，进入 socket/TCP 部分。

#### 1. 客户端创建 socket

客户端进程会调用系统调用：

```c
scoket(AF_INET, SOCKET_STREAM, 0);
```

这个系统调用进入内核后，内核会为这个 socket 创建一系列数据结构，比如：

- socket 对象
- 发送缓冲区
- 接收缓冲区
- TCP 控制块

此时还没有连上服务端。

#### 2. 客户端调用 `connect`

客户端随后调用：

```c
connect(sockfd, server_addr, ...);
```

这又是一次系统调用，内核开始做几件事情：

##### a. 选择本地端口

如果应用没有手动绑定，内核会自动分配一个临时端口。

于是连接四元组出现了：

- 源 IP：客户端 IP
- 源端口：51234
- 目的 IP：服务端 IP
- 目的端口：80

##### b. 构造 SYN 包

内核 TCP 协议栈会生成一个 TCP 报文段，带上 SYN = 1，初始序列号 seq = x，然后交给 IP 层、链路层，再由网卡发出去。

##### c. `connect` 的行为

如果是阻塞 socket，`connect` 一般会阻塞，直到握手成功或者失败；如果是非阻塞 socket，可能立即返回，后续通过时间通知连接结果。

### 服务端监听连接

在客户端发 `connect` 之前或者同时，服务端通常已经准备好了。

#### 1. 服务端创建 socket 

服务端程序先调用 

```c
socket(AF_INET, SOCK_STREAM, 0)
```

#### 2. 服务端 `bind`

```c
bind(listenfd, 0.0.0.0:80)
```

把这个 socket 绑定到某个本地 IP 和端口。

比如：

- 监听 80 端口
- 接受发到本机这个端口的 TCP 连接请求

#### 3. 服务端 `listen`

```c
listen(listenfd, backlog)
```

调用后，这个 socket 从普通 socket 变成了监听 socket。

内核会为它维护连接队列，常见地可以理解为：

- 半连接队列
- 全连接队列

这时服务端程序通常会继续调用：

```c
accept(listenfd, ...)
```

在阻塞模式下，它会睡在哪里等新连接到来。

### TCP 三次握手

现在客户端 SYN 发到了服务端。

#### 第一次握手：客户端 -> 服务端 SYN

客户端内核发送：

- SYN = 1
- seq = x

这个包经过网络到达服务端网卡。

网卡把包交给内核网络协议栈。

服务端内核发现：

- 目的端口是 80
- 本机有个监听 socket 正在监听 80

于是它知道这是一个新连接请求。

#### 第二次握手：服务端 -> 客户端 SYN+ACK

服务端内核会回复一个包：

- SYN = 1
- ACK = 1
- ack = x + 1
- seq = y

并把这个连接先放入内核管理结构里，此时连接处于半建立状态。

注意：

> 这一步通常不需要应用进程先醒来处理。**TCP 握手主要是内核自动完成的。**

#### 第三次握手：客户端 -> 服务端 ACK

客户端收到 SYN+ACK 后，客户端内核回复：

- ACK = 1
- ack = y + 1

发给服务端。

服务端内核收到后，TCP 连接正式建立。

这时这个连接会进入**全连接队列**，等待被应用进程 `accept` 取走。

### 服务端 `accept` 返回一个已连接 socket

服务端程序如果正在阻塞调用：

```c
accept(listenfd, ...)
```

当全连接队列里有新连接时，`accept` 就会返回。

返回的不是监听 socket 本身，而是一个新的 socket，记作：

```c
connfd
```

这个 `connfd` 代表：

> 这个客户端与当前服务端之间的这条 TCP 连接。

从这一刻开始：

- `listenfd` 继续负责监听新连接
- `connfd` 专门负责这条连接的数据收发

所以服务端的真实情况是：

- 一个监听 socket
- 多个已连接 socket

### 客户端发送 HTTP 请求报文

TCP 连接建好之后，客户端应用层才真正开始发送 HTTP 请求内容。

例如 HTTP/1.1 请求报文可能是：

```http
GET /api/user HTTP/1.1
Host: example.com
User-Agent: curl/8.0
Accept: */*
```

客户端应用会调用：

```c
write(sockfd, http_request, len)
```

或者 `send()`。

这也是系统调用。

#### `write/send` 时内核做了什么

客户端内核收到系统调用后，会：

1. 把用户空间的 HTTP 请求字节拷到 socket 发送缓冲区
2. TCP 协议栈根据 MSS、拥塞窗口、对端窗口等把数据分段
3. 给每段加上 TCP 头、IP 头
4. 交给网卡发出去

所以 HTTP 本身并不直接上网飞过去，而是：

> HTTP 报文先作为应用层字节流写入 TCP socket，再由 TCP/IP 协议栈封装后发送。

### 服务端接收 HTTP 请求报文

#### 1. 请求包到达服务端网卡

客户端发出的 TCP 数据包到达服务端机器。

#### 2. 网卡把数据交给内核

通常通过 DMA 把数据放到内核内存，再触发中断或由内核轮询机制处理。

#### 3. 内核协议栈处理 TCP 包

内核会检查：

- 校验和
- 序号是否正确
- 是否乱序
- 是否需要重组
- 属于哪个连接四元组

找到对应的 `connfd` 背后的 TCP 控制块后，把 **TCP payload** 放入这个 socket 的接收缓冲区。

这里放进去的是：

- HTTP 请求报文的字节流
- 不是 TCP 头本身

此时，这个 socket 就可读了。

### 服务端应用读取 HTTP 请求

接下来服务端程序才真正去读。

它可能有几种模式。

#### 模式 A：阻塞 `read`

服务端线程执行：

```c
read(connfd, buf, size)
```

##### 情况 1：数据已经到了

内核发现接收缓冲区里有数据，于是：

- 把数据从内核 socket 接收缓冲区拷到用户态 `buf`
- `read` 返回读取字节数

##### 情况 2：数据还没到

如果 socket 是阻塞的：

- 线程睡眠
- 等到数据到达后内核唤醒它
- 再执行拷贝并返回

这就是同步阻塞 IO。

#### 模式 B：非阻塞 `read`

如果 socket 是非阻塞的：

- 没数据：立刻返回 `EAGAIN`
- 线程自己决定稍后再来读

#### 模式 C：`epoll + read`

更常见于高并发服务器。

流程是：

1. `epoll` 先监听这个 `connfd`
2. 当内核发现其接收缓冲区有数据时，通知应用“可读”
3. 线程再对这个 `connfd` 调 `read`

这时线程不需要傻傻不停 `read` 轮询所有连接。

### HTTP 协议解析

服务端应用拿到字节流后，开始解析 HTTP。

例如解析出：

- 方法：GET
- 路径：`/api/user`
- 版本：HTTP/1.1
- Header：Host、Cookie、Authorization...
- Body：如果是 POST/PUT 还有请求体

注意这里有一个重要点：

> TCP 只保证字节流，不保证一个 `read` 就读到一个完整 HTTP 请求。

比如一个 HTTP 请求可能：

- 一次 `read` 只读到一半
- 或一次 `read` 读到多个请求（长连接/粘包场景）

所以 HTTP 服务器要自己根据协议规则判断：

- 请求头是否完整
- Content-Length 是否收齐
- chunked body 是否结束

### 业务处理 + 响应返回

服务端 HTTP 框架解析完后，就会进入你的业务代码，比如：

- 路由匹配 `/api/user`
- 查数据库
- 调 RPC
- 拼装 JSON

然后调用 `write/send` 把 HTTP 响应发回客户端。

这个过程和请求读取相反：

- 用户态响应数据拷到内核发送缓冲区
- TCP 分段发出
- 客户端内核接收
- 客户端应用再 `read`

在整个 HTTP 过程中

**用户态应用负责**

客户端：

- 解析 URL
- 调 `socket/connect/write/read`

服务端：

- 调 `socket/bind/listen/accept/read/write`
- 解析 HTTP
- 执行业务逻辑

**内核负责**

- socket 对象管理
- TCP 三次握手/四次挥手
- 重传、确认、排序
- 接收缓冲区/发送缓冲区
- `accept/read/write` 等系统调用处理
- 唤醒/阻塞线程
- `select/poll/epoll` 事件通知

**网卡/硬件负责**

- 真正发出和接收比特流
- DMA 搬运数据到内存

这下我们就明白了为什么会出现网络 IO。我们平常使用 Spring MVC 写的服务端时如此的简单。但在 MVC 的背后，涉及到了非常多的步骤。 

HTTP 作为一个应用层协议，它是通过 TCP 建立的网络通道中传输数据的。一个 `http://example.com:8080/api/user` 地址，TCP 层会用协议，端口等信息确定四元组，而 `/api/user` 就是 HTTP 层面的内容了，HTTP 应层通过 `/api/user` 唯一确定服务端的资源。

`select`、`poll`、`epoll` 都是 IO 多路复用机制，用于让单个线程同时监听多个文件描述符，并在其就绪时进行处理。
 `select` 使用位图保存 fd 集合，存在 fd 数量限制，并且每次调用都需要重新传递整个集合，内核和用户态都需要线性遍历全部 fd，因此在大量连接场景下性能较差。
 `poll` 使用数组描述 fd，没有 `select` 的固定数量限制，但本质上仍然需要每次遍历全部 fd，因此性能问题与 `select` 类似。
 `epoll` 是 Linux 下高效的多路复用机制，通过 `epoll_ctl` 将 fd 预先注册到内核，由内核维护监听集合和就绪队列，`epoll_wait` 只返回已经就绪的 fd，因此特别适合大量连接但同时活跃连接较少的高并发场景。
 需要注意的是，`epoll` 优化的是事件通知机制，而不是消除 `read/write` 时的数据拷贝；它通常仍属于同步 IO 模型中的 IO 多路复用。
