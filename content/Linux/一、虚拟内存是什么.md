**操作系统给每个进程“伪造”出来的一套独立、连续、很大的内存视图。**

程序以为自己在直接操作一大片连续内存，
 但实际上它访问的是**虚拟地址**，最后由硬件和操作系统把它映射到真正的**物理内存**，必要时甚至映射到磁盘。

------

# 一、虚拟内存是什么

假设机器只有 8GB 物理内存，但你开了很多进程。
 如果没有虚拟内存，程序就得直接操作真实物理地址，比如“0x12345678 这个物理位置”。

这会有几个大问题：

1. **不同进程会互相踩内存**
2. 程序必须关心自己被装载到哪个物理位置，很麻烦
3. 物理内存不连续时，分配会很难
4. 程序能用的内存大小，完全被真实 RAM 直接限制

所以 OS 引入了虚拟内存：

- 每个进程都看到自己的地址空间
- 看起来像一整块连续内存
- 不同进程的同一个虚拟地址，实际上可以映射到不同物理地址
- 某些暂时不用的数据，还可以先不放进物理内存

所以：

**虚拟内存 = 虚拟地址空间 + 地址映射机制 + 缺页调页机制**

------

# 二、虚拟内存解决了什么问题

## 1）进程隔离

进程 A 和进程 B 都可以有地址 `0x1000`，但它们实际对应的物理地址完全不同。

所以：

- A 不能直接访问 B 的内存
- 一个进程崩了，不会随便把别的进程内存写坏

这就是内存隔离。

------

## 2）让程序看到“连续内存”

程序申请一段 100MB 内存时，它看到的是连续虚拟地址。
 但底层物理内存完全可以是零散的几块页，不需要真的连续。

这大大降低了内存管理难度。

------

## 3）让“可用内存”看起来比物理内存更大

虚拟内存允许：

- 一部分页面在 RAM
- 一部分页面暂时在磁盘（swap / page file / 文件映射）

所以进程可以拥有一个很大的虚拟地址空间，哪怕物理内存没那么大。

注意这不是“凭空变大”，而是：

**不常用的内容先放磁盘，需要时再换回来。**

------

## 4）支持共享

多个进程可以把不同虚拟地址映射到同一个物理页。

典型例子：

- 多个进程共享同一份动态链接库代码段
- 父子进程 `fork` 后先共享页，写时再复制（COW）

------

# 三、虚拟内存是如何实现的（核心机制）

实现虚拟内存，靠的是三样东西：

1. **CPU 硬件（MMU）**
2. **页表（Page Table）**
3. **操作系统内核**

------

## 1）虚拟地址和物理地址

程序里写的地址是**虚拟地址**。
 真正访问内存芯片时，用的是**物理地址**。

中间需要做一次转换：

**虚拟地址 -> 物理地址**

这个转换由 CPU 里的 **MMU（内存管理单元）** 负责。

------

## 2）分页（Paging）

现代系统基本都用“页”来管理虚拟内存。

把内存分成固定大小的小块：

- 虚拟内存分成：**虚拟页**
- 物理内存分成：**物理页框（page frame）**

常见页大小：

- 4KB（最常见）
- 2MB / 1GB（大页）

------

### 地址怎么拆

例如一个虚拟地址：

- 高位：表示它属于哪个**页**
- 低位：表示页内的**偏移量**

举例（假设页大小 4KB）：

- 4KB = 2^12
- 那么虚拟地址的低 12 位就是页内偏移
- 剩下高位就是虚拟页号

MMU 做的事就是：

1. 先拿虚拟页号查页表
2. 找到对应物理页框号
3. 再加上原来的页内偏移
4. 得到最终物理地址

所以本质上：

**物理地址 = 物理页框基址 + 页内偏移**

------

## 3）页表（Page Table）

页表就是“映射关系表”。

它记录：

- 某个虚拟页，对应哪个物理页框
- 这个页是否存在
- 是否可读/可写/可执行
- 是否被访问过
- 是否被修改过（脏页）

你可以把它理解成：

| 虚拟页号 | 物理页框号 | 权限 | 是否在内存 |
| -------- | ---------- | ---- | ---------- |
| 0x123    | 0x88A      | R/W  | 是         |
| 0x124    | -          | -    | 否（缺页） |

------

## 4）MMU：地址翻译的执行者

程序执行一条指令，比如访问：

```
*0x7fff12345678
```

CPU 不会直接把这个当成物理地址打到内存总线上。
 它会先交给 MMU 翻译。

流程大致是：

1. CPU 产生虚拟地址
2. MMU 查页表
3. 若找到有效映射 -> 得到物理地址
4. 去访问真正的物理内存

------

## 5）TLB：加速页表查询

如果每次内存访问都去查页表，那会很慢。
 因为你访问一个变量都要先多查一次甚至多次内存。

所以 CPU 里有个缓存，叫：

**TLB（Translation Lookaside Buffer）**

它缓存最近用过的“虚拟页 -> 物理页框”的映射。

所以平时流程通常是：

1. 先查 TLB
2. 命中：直接拿到物理地址，快
3. 未命中：再去查页表
4. 查到后把结果写回 TLB

这就是为什么 TLB 很重要。

------

# 四、如果页不在内存里，会发生什么（缺页）

这是虚拟内存最关键的一步。

如果程序访问一个虚拟页，但这个页当前**不在物理内存**：

- MMU 查页表发现该页无效 / 不在内存
- CPU 会触发一个异常：**缺页异常（page fault）**
- 然后陷入内核

------

## 缺页后的流程

操作系统接手后会做这些事：

### 1）判断这次访问是否合法

比如：

- 这个地址是否属于进程的合法虚拟空间？
- 是否越界？
- 是否访问了无权限的页？

如果不合法：

- 直接给进程一个错误（如段错误）

如果合法：

- 继续调页

------

### 2）找到这个页的数据来源

页的数据可能来自：

- 可执行文件 / 动态库
- 内存映射文件（mmap）
- 交换区（swap）
- 新申请的匿名页（初始清零）

------

### 3）找一个空闲物理页框

如果有空闲页框，直接用。
 如果没有，就得先淘汰一个已有页面。

这就涉及**页面置换算法**，比如近似 LRU。

------

### 4）必要时把旧页写回磁盘

如果被淘汰页是脏页（被修改过）：

- 需要先写回磁盘

如果没修改过：

- 可以直接丢掉

------

### 5）把目标页从磁盘读入物理内存

把真正需要的那一页读到物理页框中。

------

### 6）更新页表

告诉 MMU：

- 这个虚拟页现在对应哪个物理页框
- 标记为“在内存中”
- 设置权限位

------

### 7）重新执行刚才那条指令

CPU 回到用户态，再执行刚才访问那块内存的指令。
 这次因为页已经进来了，就能成功访问。

------

# 五、这就是“按需加载”（Demand Paging）

程序启动时，并不会把它所有代码、数据、堆都一次性全装入内存。
 否则太慢也太浪费。

而是：

- 先建立虚拟地址空间
- 真正访问到哪一页，再把哪一页调进来

这叫：

**按需分页 / 请求分页（Demand Paging）**

所以很多程序看起来“占用内存很大”，其实只是虚拟地址空间大，未必都真的进了物理内存。

------

# 六、一个具体例子：程序访问变量时发生了什么

假设你的程序访问一个变量，虚拟地址是：

```
0x12345678
```

假设页大小是 4KB。

------

## 第一步：拆地址

- 虚拟页号 = 高 20 位（简化理解）
- 页内偏移 = 低 12 位

------

## 第二步：查 TLB

- 如果 TLB 里有这页映射，直接得到物理页框号
- 如果没有，查页表

------

## 第三步：查页表

假设查到：

- 虚拟页 `0x12345`
- 对应物理页框 `0x9AB`

------

## 第四步：拼出物理地址

物理地址 = 物理页框基址 + 偏移

于是最终可能变成：

```
0x9AB678
```

CPU 再去访问真正的 RAM。

------

## 如果页表说“不在内存”

那就触发缺页，交给内核处理，调页后再重试。

------

# 七、页表本身怎么组织

如果虚拟地址空间很大，页表会非常大。
 所以现代系统不会只用一张简单的“大数组页表”，而是常用：

**多级页表**

比如：

- 一级页表
- 二级页表
- 三级页表
- 四级页表

64 位系统里常见四级页表甚至更多层次。

这样做的原因是：

- 虚拟空间很大，但并不是每一段都真的用到了
- 多级页表可以按需分配，节省内存

代价是：

- TLB 未命中时，页表遍历更复杂
- 但整体更省空间

------

# 八、虚拟内存不只是“分页”，还包括权限控制

页表项里不只有地址映射，还有权限位：

- 可读（R）
- 可写（W）
- 可执行（X）
- 用户态可访问 / 仅内核可访问

所以虚拟内存还承担了**保护**作用。

例如：

- 代码段通常可读可执行，不可写
- 数据段可读可写
- 内核页用户态不可访问

这就是为什么非法访问内存会崩。

------

# 九、写时复制（COW）也是虚拟内存的重要应用

比如 `fork()` 创建子进程时，
 父子进程一开始看起来像有一份完整拷贝。

但 OS 不会立刻真的把所有物理页复制一遍，那太贵了。

它会：

1. 父子进程先共享同一批物理页
2. 都标记成只读
3. 谁先写某一页，谁触发缺页异常
4. 内核再真正复制该页
5. 让写的一方使用新页

这叫：

**Copy-On-Write（写时复制）**

这本质上也是虚拟内存机制在发挥作用。

------

# 十、为什么虚拟内存会有性能开销

虚拟内存很强大，但不是免费的。

------

## 1）地址翻译开销

每次访问内存都要做地址翻译。
 虽然有 TLB 缓存，但 TLB miss 仍然贵。

------

## 2）缺页异常很贵

如果页不在内存：

- 进入内核
- 可能磁盘 I/O
- 更新页表
- 再返回

这会非常慢，比普通内存访问慢几个数量级都不止。

------

## 3）频繁换页会抖动（Thrashing）

如果物理内存太小、进程工作集太大，系统可能不断：

- 换出页
- 再换入页
- 再换出别的页

CPU 大量时间耗在调页，而不是做真正业务。

这就是“抖动”。

------

# 十一、虚拟内存和“内存不够”是什么关系

很多人会误解成：

> 虚拟内存 = 把磁盘当内存，所以内存就无限了

这是不对的。

更准确地说：

- 虚拟内存让程序拥有大地址空间
- 磁盘只是“后备存储”
- 当 RAM 不够时，系统可以把不常用页挪到磁盘

但磁盘远慢于内存，所以：

**虚拟内存让程序“能跑”，不代表“跑得快”。**

------

# 十二、你可以这样记住它的本质

虚拟内存本质上做了三件事：

## 1）欺骗程序

让每个程序都觉得自己有一套独立、连续的大内存。

## 2）映射地址

通过页表和 MMU，把虚拟地址翻译成物理地址。

## 3）按需调页

不常用的页可以不放在 RAM，访问时再调入。

------

# 十三、一句话总结

**虚拟内存就是操作系统和硬件共同实现的一套“地址映射 + 内存隔离 + 按需调页”机制：程序使用虚拟地址，MMU 通过页表把它映射到物理内存；如果目标页不在内存，就触发缺页异常，由操作系统把页面调入。**
